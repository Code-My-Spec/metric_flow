## Issues

### OAuth connect button absent when provider credentials not configured

#### Severity
MEDIUM

#### Scope
APP

#### Description
On both the connected and not-connected detail views (`/integrations/connect/:provider`), the `[data-role="oauth-connect-button"]` element is only rendered when `@authorize_url` is non-nil. When OAuth credentials are not configured for a provider, `Integrations.authorize_url/1` returns an error and `@authorize_url` is set to nil.

For not-connected platforms, instead of a connect button, an alert warning is shown: "OAuth is not configured for this provider. Please contact your administrator." For connected platforms, the message "Reconnect is not available — this provider is not currently configured for OAuth." is shown.

While this is technically correct behavior, it means users with unconfigured providers see a warning message with no action path. The spec requires the `oauth-connect-button` to be present for initiating OAuth — this element is the primary CTA for the feature.

Reproduced at: `http://localhost:4070/integrations/connect/google_analytics` (not connected) and `http://localhost:4070/integrations/connect/google_ads` (connected).

### error_description ignored when error is access_denied

#### Severity
LOW

#### Scope
APP

#### Description
When the OAuth callback receives `?error=access_denied&error_description=User+denied+access`, the `format_oauth_error/3` function in `IntegrationOauthController` matches the more specific `access_denied` clause first, ignoring the `error_description`. The user sees "Access was denied. Please try again if you want to connect." instead of a message that incorporates the provider-supplied description.

The clause ordering means the `error_description` pattern (`%{"error" => error, "error_description" => description}`) is never reached for `access_denied` errors. The provider-supplied description is silently discarded.

Reproduced at: `http://localhost:4070/integrations/oauth/callback/google_ads?error=access_denied&error_description=User+denied+access`

### Unknown "google" integration platform card appears on selection page

#### Severity
LOW

#### Scope
APP

#### Description
The platform selection page at `/integrations/connect` shows a fifth platform card with `data-platform="google"` labeled "Google" (description: "Google Ads, Analytics, and Search Console"). This is not in the canonical platforms list (`@canonical_platforms`) but appears because a legacy integration with `provider: :google` exists in the QA database.

The `build_platform_list/1` function unions canonical keys, configured provider keys, and existing integration keys — meaning any integration in the database will produce a card even if the provider is not recognized. The "Google" card has a "Reconnect" button but clicking it would redirect to `/integrations/oauth/google` which is likely not a configured provider and would fail.

This is a data artifact from previous QA runs and is also tracked in the existing issue qa-435 (pre-existing integrations in QA database).
