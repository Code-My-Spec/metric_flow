## Issues

### Visiting /integrations/connect/:provider with an unknown atom crashes with Ecto.Query.CastError

#### Severity
HIGH

#### Description
Navigating to `http://localhost:4070/integrations/connect/unknown_provider` returns a 500 error page with `Ecto.Query.CastError`. The LiveView's `handle_provider_params/2` rescues `ArgumentError` from `String.to_existing_atom/1` to handle unknown providers, but the atom `:unknown_provider` already exists in the BEAM (likely defined as a module attribute or used elsewhere), so the rescue does not fire. Instead, the atom is passed to `Integrations.get_integration/2` which queries the database with a value that cannot be cast to the Integration provider Ecto.Enum, resulting in an unhandled crash.

The fix should include a guard that checks whether the provider atom is a member of the allowed Ecto.Enum values, or catch `Ecto.Query.CastError` in addition to `ArgumentError`.

### oauth-connect-button absent for Google Ads and Google Analytics in dev environment

#### Severity
MEDIUM

#### Scope
QA

#### Description
The `oauth-connect-button` element is not rendered for Google Ads or Google Analytics on their detail pages (`/integrations/connect/google_ads`, `/integrations/connect/google_analytics`) because `Integrations.authorize_url/1` returns `{:error, _}` when OAuth credentials are not configured. The LiveView correctly falls back to a warning: "OAuth is not configured for this provider. Please contact your administrator."

Scenario 3 of the brief expects the button to be present with an `href` pointing to an OAuth authorization URL. This expectation cannot be verified in the dev environment without Google OAuth credentials.

Facebook Ads OAuth IS configured (credentials present in dev) and its `oauth-connect-button` works correctly with a valid `href`.

To fully test Scenario 3 for all platforms, Google OAuth credentials (GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET) would need to be configured in the dev environment.

### OAuth callback with code returns misleading error "This platform is not yet supported"

#### Severity
MEDIUM

#### Description
Visiting `http://localhost:4070/integrations/oauth/callback/google_ads?code=test_auth_code&state=test_state` results in the flash error "This platform is not yet supported" on the resulting detail page. Google Ads is a supported platform — the actual failure is that the OAuth token exchange fails because the dev environment has no real Google Ads credentials.

The error originates from `format_oauth_error/2` matching the `:unsupported_provider` error reason returned by `handle_oauth_callback/4` when the provider argument is nil. However, in this code path `provider` is not nil — the error comes from `Integrations.handle_callback/4` failing the token exchange and returning `{:error, :unsupported_provider}` or similar. The message "This platform is not yet supported" should be reserved for genuinely unsupported providers; token exchange failures should show a more accurate error like "Could not complete the connection. Please try again."

### error_description is ignored when error=access_denied in OAuth callback

#### Severity
LOW

#### Description
When the OAuth callback includes both `error=access_denied` and `error_description=...`, the `error_description` value is silently ignored. The `format_oauth_error/2` function in `IntegrationOAuthController` matches the `%{"error" => "access_denied"}` clause before the `%{"error" => error, "error_description" => description}` clause, so the description is never shown to the user.

This means if a provider sends a more specific reason for the access denial (e.g., "User is not authorized for this app"), that detail is lost and the user only sees the generic "Access was denied. Please try again if you want to connect." message.

URL tested: `http://localhost:4070/integrations/oauth/callback/google_ads?error=access_denied&error_description=User+denied+access`

### Pre-existing "Google" integration visible in platform selection grid

#### Severity
INFO

#### Scope
QA

#### Description
The platform selection grid at `/integrations/connect` shows a "Google" platform card (data-platform="google") from a pre-existing integration created in prior QA test runs. This card has no description text and does not appear in the canonical platform list defined in `@canonical_platforms`. It appears because `build_platform_list/1` includes platforms from existing user integrations even if they're no longer in the canonical list.

This is a QA data state issue — the `qa_seeds.exs` script does not clean up old integrations. Re-running the seed script does not remove pre-existing integrations, so they accumulate across test runs. The seeds script should be updated to delete existing integrations for the QA user before creating new test state.
