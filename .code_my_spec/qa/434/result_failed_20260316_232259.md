## Issues

### Connect provider routes use `google` not `google_ads` or `google_analytics`

#### Severity
HIGH

#### Scope
APP

#### Description
The spec and brief define routes for `/integrations/connect/google_ads` and `/integrations/connect/google_analytics`, and expect `data-platform="google_ads"` and `data-platform="google_analytics"` attributes on the platform selection grid.

The implementation uses a single Google OAuth provider (`data-platform="google"`) that covers both Google Ads and Google Analytics. Navigating to `/integrations/connect/google_ads` silently redirects to `/integrations/connect` because `validate_provider/1` does not recognize `google_ads` as a valid provider.

The spec says: "Google Ads", "Facebook Ads", and "Google Analytics" should each appear with `data-platform="google_ads"`, `data-platform="facebook_ads"`, and `data-platform="google_analytics"`. The actual page shows "Google" (covering both), "Facebook" (with `data-platform="facebook_ads"`), and "QuickBooks".

This is a mismatch between the spec's expected platform model and the implementation's OAuth provider model. Consumers testing against the spec-defined selectors will fail.

### Account selection page uses radio buttons, not checkboxes

#### Severity
MEDIUM

#### Scope
APP

#### Description
The BDD spec and brief expect `input[type='checkbox'][data-role='account-checkbox']` for account selection. The implementation renders `input[type='radio']` with `data-role='property-radio'`. The account selection form allows choosing a single GA4 property, not multiple accounts.

Selector: `[data-role='account-checkbox']` returns no elements. The actual data-role is `property-radio`.

### Facebook and QuickBooks accounts pages show Google Analytics-specific text

#### Severity
MEDIUM

#### Scope
APP

#### Description
Visiting `/integrations/connect/facebook_ads/accounts` renders the account selection view with the heading "Facebook — Select Accounts" but the body text says "Choose which GA4 property to sync data from." and the form field is labeled "GA4 Property ID" with the help text "Find this in Google Analytics under Admin → Property Settings". This text is incorrect for a Facebook provider.

The `render_account_selection/1` function is not provider-aware — all providers show the same Google Analytics-centric UI copy. For Facebook and QuickBooks, this text is misleading and incorrect.

### BDD spec references `google_ads` provider routes that do not exist

#### Severity
LOW

#### Scope
QA

#### Description
The brief (Scenarios 1–5, 7–8) references routes like `/integrations/connect/google_ads`, `/integrations/connect/google_ads/accounts`, and `/integrations/oauth/callback/google_ads?...`. None of these routes resolve — they all redirect to `/integrations/connect` because `google_ads` is not a recognized provider key.

The correct provider key is `google`. The spec was written against a different data model than what was implemented. Tests should be updated to use the actual provider key `google` for scenarios involving Google Ads or Google Analytics.
