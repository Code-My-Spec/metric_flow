## Issues

### OAuth connect button not rendered when provider is not configured

#### Severity
HIGH

#### Scope
APP

#### Description
On the per-platform detail view (`/integrations/connect/:provider`), the `[data-role='oauth-connect-button']` anchor is only rendered when `@authorize_url` is non-nil. In development and QA environments, `Integrations.authorize_url/1` returns `{:error, _}` for all providers (Google Ads, Facebook Ads, Google Analytics) because OAuth credentials are not configured. As a result, the Connect button is never shown — instead the page shows "Reconnect is not available — this provider is not currently configured for OAuth."

This means a real user visiting the connect page in a production environment without configured OAuth credentials also sees no connect button, with no clear guidance on why. The error state should either show a more helpful message or the admin should configure OAuth before the connect flow is presented to users.

Reproduced at: `http://localhost:4070/integrations/connect/google_ads` and `http://localhost:4070/integrations/connect/google_analytics`

### OAuth callback is a controller redirect, not a LiveView result page

#### Severity
MEDIUM

#### Scope
APP

#### Description
The spec (`connect.spec.md`) describes the OAuth callback as a LiveView at `/integrations/oauth/callback/:provider` that renders either a "Integration Active" success view or a "Connection Failed" error view with dedicated navigation links ("View Integrations", "Try again", "Back to integrations"). The actual implementation uses `IntegrationOAuthController` which immediately redirects to `/integrations/connect/:provider` with a flash message.

As a result:
- There is no transient "Connecting your account..." loading state
- There is no "Integration Active" confirmation page with "View Integrations" button
- There is no dedicated "Connection Failed" error page with "Try again" link
- Error feedback is a flash banner on the detail page, not a full error view

The spec describes a better UX than what is implemented. The redirect-with-flash approach works functionally but does not meet the design specification.

Reproduced at: `http://localhost:4070/integrations/oauth/callback/google_ads?error=access_denied` (redirects to `/integrations/connect/google_ads` with flash)

### Pre-existing "google" integration appears as unknown platform card

#### Severity
LOW

#### Scope
APP

#### Description
The platform selection page at `/integrations/connect` shows a card with `data-platform="google"` and an empty description alongside the canonical platforms. This appears because there is a legacy integration with provider key `:google` in the QA database. The `build_platform_list/1` function includes integration providers not in the canonical list, causing this orphaned card to appear.

The card shows provider key "Google" (derived from the atom), has an empty description, shows "Connected" status, and has a "Reconnect" button with `phx-value-provider="google"`. Clicking that button would attempt to initiate OAuth for the undefined `:google` provider, which would fail.

Reproduced at: `http://localhost:4070/integrations/connect` when logged in as `qa@example.com`

### Pre-existing integrations in QA database cause brief's "clean state" assumptions to fail

#### Severity
LOW

#### Scope
QA

#### Description
The brief states "The seed creates a QA owner user with no pre-existing integrations, which matches the expected pre-OAuth state for this story." However, the QA database already has integrations for Facebook Ads, Google, and QuickBooks connected to `qa@example.com`. The seed script does not clear existing integrations before creating them.

This causes Scenario 2 (which asserts "no Re-authenticate text") and Scenario 6 (which asserts "Integration saved does not appear") to operate against pre-connected state rather than the expected fresh state. The brief should either instruct QA to clear existing integrations, or the seed script should remove existing integrations for QA accounts before seeding.
