## Issues

### google_ads is not a registered provider — seed integration never renders on integrations page

#### Severity
MEDIUM

#### Scope
QA

#### Description
The seed script (`priv/repo/qa_seeds.exs`) creates a `google_ads` integration for `qa@example.com` with `provider_metadata` containing `"selected_accounts": ["Campaign Alpha", "Campaign Beta"]`. However, `Integrations.list_providers()` returns only `[:google, :facebook_ads, :quickbooks]` — the atom `:google_ads` is not registered.

The integrations index builds its platform list exclusively from `list_providers()`, so the google_ads integration record exists in the database but is never rendered on the page. Scenario 6 (selected accounts) and any scenario requiring a "Google Ads" named card cannot be verified.

Fix options: (1) Add `:google_ads` to `@default_providers` in `lib/metric_flow/integrations.ex`, or (2) Update the seed to create a `google` integration with selected_accounts, since `:google` is a registered provider.

### Disconnect confirmation uses inline panel, not modal overlay

#### Severity
LOW

#### Description
Clicking "Disconnect" on a connected integration card expands an inline warning panel (`[data-role='disconnect-warning']`) within the card rather than opening a `.modal` overlay. The spec's Design section describes a "Disconnect confirmation modal" with a `.modal` overlay element. No modal element or `modal-open` class exists in the rendered HTML.

The inline panel is fully functional and provides the warning text and Confirm/Cancel buttons. The discrepancy is between the spec's prescribed implementation and the actual implementation.

Reproduced at: `http://localhost:4070/integrations` — click "Disconnect" on any connected card.

### Confirm disconnect button labeled "Confirm" instead of "Disconnect"

#### Severity
LOW

#### Description
The `[data-role='confirm-disconnect']` button inside the disconnect warning panel displays "Confirm" rather than "Disconnect". The spec's Design section states the confirm button should be labeled "Disconnect" to make the destructive action explicit. Users may expect to see the word "Disconnect" on the final confirmation step.

Source: `lib/metric_flow_web/live/integration_live/index.ex` line 161 — `"Confirm"`.

Reproduced at: `http://localhost:4070/integrations` — click "Disconnect", then observe the button label in the panel.

### Disconnect flash message missing "no new data will sync" clause

#### Severity
LOW

#### Description
After confirming disconnect, the flash message reads: "Disconnected from {Platform}. Historical data is retained."

The spec and brief indicate the message should include an additional clause: "no new data will sync after disconnecting." or equivalent. The omission leaves users without a clear statement about the consequence for future data.

Source: `lib/metric_flow_web/live/integration_live/index.ex` — `handle_event("disconnect", ...)` puts `"Disconnected from #{name}. Historical data is retained."`.

Reproduced at: `http://localhost:4070/integrations` — disconnect any connected integration and observe the flash.

### Connect button initiates OAuth immediately instead of showing informational message

#### Severity
INFO

#### Description
In the Available Platforms section, clicking the "Connect" button (`[data-role='reconnect-integration']`) immediately redirects to the OAuth provider authorization page. The brief expected a flash message such as "Reconnect {Platform}: authorize your account on the Connect page." before initiating OAuth. No flash is shown.

The direct OAuth redirect is not necessarily wrong UX — it reduces friction — but it diverges from the brief's specified behavior. If the intent is to keep users on the integrations page with an informational prompt first, the `handle_event("initiate_connect", ...)` handler needs to change from `redirect` to `put_flash` + stay on page.

Source: `lib/metric_flow_web/live/integration_live/index.ex` — `handle_event("initiate_connect", ...)` calls `redirect(socket, to: ~p"/integrations/oauth/#{provider_str}")`.

Reproduced at: `http://localhost:4070/integrations` — disconnect an integration, then click "Connect" in Available Platforms.
