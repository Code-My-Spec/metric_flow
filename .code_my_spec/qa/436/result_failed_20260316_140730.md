## Issues

### Disconnect modal heading does not include platform name

#### Severity
LOW

#### Scope
APP

#### Description
The disconnect confirmation modal heading reads "Confirm Disconnect" rather than the spec-required "Disconnect {Platform Name}?" pattern (e.g., "Disconnect Google?"). The modal body paragraph does include the platform name in the text, but the `<h3>` heading is generic.

Source: `lib/metric_flow_web/live/integration_live/index.ex` — `<h3 class="font-bold text-lg">Confirm Disconnect</h3>`.

Reproduced at: `http://localhost:4070/integrations`, click "Disconnect" on any connected platform.

### Disconnect warning paragraph missing data-role attribute

#### Severity
LOW

#### Scope
APP

#### Description
Scenario 10 checks for `[data-role='disconnect-warning']` to verify the warning message is present. The modal body uses a plain `<p class="py-4">` tag with no `data-role` attribute. The warning text is correct and fully present, but the element cannot be identified by the specified selector. The spec requires `data-role="disconnect-warning"` for this paragraph.

Fix: add `data-role="disconnect-warning"` to the `<p class="py-4">` element in the modal body.

Source: `lib/metric_flow_web/live/integration_live/index.ex` — disconnect modal `<p class="py-4">` has no `data-role`.

### Confirm button in disconnect modal labeled "Confirm" instead of "Disconnect"

#### Severity
LOW

#### Scope
APP

#### Description
The spec and brief both specify the confirmation button text should be "Disconnect". The actual button renders "Confirm". The button has the correct `data-role="confirm-disconnect"` and `phx-click="disconnect"` handler, but the visible label does not match the spec.

Source: `lib/metric_flow_web/live/integration_live/index.ex` — `<button ... class="btn btn-error">Confirm</button>`.

Reproduced at: `http://localhost:4070/integrations`, click "Disconnect" on any connected platform to open the modal.

### Disconnect flash message missing "no new data will sync" clause

#### Severity
LOW

#### Scope
APP

#### Description
After confirming disconnect, the flash message reads: "Disconnected from {platform name}. Historical data is retained."

The spec requires: "Disconnected from {platform name}. Historical data is retained; no new data will sync after disconnecting."

The second clause is omitted from the post-disconnect flash. The modal body does include equivalent text, but the flash does not.

Source: `lib/metric_flow_web/live/integration_live/index.ex` — `put_flash` in `handle_event("disconnect", ...)`, line ~327.

### Connect button initiates OAuth redirect instead of showing informational flash

#### Severity
LOW

#### Scope
APP

#### Description
In the Available Platforms section, clicking "Connect" (`[data-role='reconnect-integration']`) immediately redirects the browser to the OAuth provider's authorization page. The brief expected a flash message "Reconnect {Platform}: authorize your account on the Connect page." to keep the user on the integrations page first.

The `handle_event("initiate_connect", ...)` handler calls `redirect(socket, to: ~p"/integrations/oauth/#{provider_str}")` which triggers immediate OAuth without any intermediate user feedback.

Verified: clicking Facebook Ads "Connect" redirected to `https://www.facebook.com/login.php?...` with no flash shown.

Source: `lib/metric_flow_web/live/integration_live/index.ex`, `handle_event("initiate_connect", ...)`.

### Connected date format is YYYY-MM-DD instead of human-readable format

#### Severity
INFO

#### Scope
APP

#### Description
The `[data-role='integration-connected-date']` element displays dates as "Connected 2026-03-16" using `Calendar.strftime(integration.inserted_at, "%Y-%m-%d")`. The brief expected a human-readable format like "Connected Mar 16, 2026".

Source: `lib/metric_flow_web/live/integration_live/index.ex` — `Calendar.strftime(integration.inserted_at, "%Y-%m-%d")`.
