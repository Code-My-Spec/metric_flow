### Connect button initiates OAuth redirect instead of staying on integrations page

#### Severity
LOW

#### Scope
APP

#### Description
In the Available Platforms section, clicking the "Connect" button (`[data-role='reconnect-integration']`) immediately redirects the browser to the OAuth provider's authorization page (e.g., `https://accounts.google.com/...`). The spec design section describes the button as initiating connection, but the brief expected a flash message "Reconnect {Platform}: authorize your account on the Connect page." to keep the user on the integrations page.

The `handle_event("initiate_connect", ...)` handler calls `redirect(socket, to: ~p"/integrations/oauth/#{provider_str}")` which triggers immediate OAuth. Whether a flash message is preferred over direct OAuth redirect is a product decision, but the current behavior contradicts the brief specification.

Source: `lib/metric_flow_web/live/integration_live/index.ex`, line 296-298.

Reproduced at: `http://localhost:4070/integrations`, click "Connect" on any available platform.

### Seed script for google_ads integration cannot run — provider not registered and tokens encrypted

#### Severity
MEDIUM

#### Scope
QA

#### Description
The brief's seed command attempts to create a `google_ads` integration via `mix run -e`, but this fails for two reasons:

1. `:google_ads` is not a registered provider in the application. `Integrations.list_providers()` returns `[:google, :facebook_ads, :quickbooks]`. An integration inserted with provider `:google_ads` would not appear in the integrations UI.

2. The `access_token` and `refresh_token` fields on `Integration` use `MetricFlow.Encrypted.Binary` — plain string values cannot be inserted via `Integration.changeset/2` without the encryption adapter configured for direct database access (the `mix run --no-start` approach does not load the full application stack needed for encrypted field support).

The brief should be updated to use `:google` as the provider and to reference a `.exs` seed script in `priv/repo/` that handles the encryption properly (or documents that selected-accounts testing requires a live integration created via the OAuth flow).

### Integration sync status element empty at rest — brief selector incorrect

#### Severity
INFO

#### Scope
QA

#### Description
The brief (Scenario 5) checks for `.badge-success` inside `[data-role='integration-sync-status']`. The actual implementation renders `[data-role='integration-sync-status']` as an empty `<span>` at rest — the "Connected" badge (`<span data-status="connected" class="badge badge-success">Connected</span>`) is a sibling element, not a child of the sync status span.

The correct selector to verify "Connected" status is `[data-status='connected'].badge-success` or `[data-role='integration-card'][data-status='connected'] .badge-success`. The brief's Scenario 5 assertion needs to be updated to match the actual DOM structure.
