## Issues

### Disconnect modal heading does not include platform name

#### Severity
LOW

#### Scope
APP

#### Description
The disconnect confirmation modal heading reads "Confirm Disconnect" rather than the spec-required "Disconnect {Platform Name}?" pattern (e.g., "Disconnect Google?"). The modal body paragraph does include the platform name ("Are you sure you want to disconnect Google?"), but the `<h3>` heading is generic.

Source: `lib/metric_flow_web/live/integration_live/index.ex` — `<h3 class="font-bold text-lg">Confirm Disconnect</h3>`.

Reproduced at: `http://localhost:4070/integrations`, click "Disconnect" on any connected platform.

### Disconnect warning paragraph missing data-role attribute

#### Severity
LOW

#### Scope
APP

#### Description
Scenario 10 checks for `[data-role='disconnect-warning']` to verify the warning message is present. The actual modal body uses a plain `<p>` tag with no `data-role` attribute. The warning text itself is correct and fully present, but the element cannot be identified by the specified selector.

Fix: add `data-role="disconnect-warning"` to the `<p class="py-4">` element in the modal.

Source: `lib/metric_flow_web/live/integration_live/index.ex` — modal `<p class="py-4">` has no `data-role`.

### Confirm button in disconnect modal labeled "Confirm" not "Disconnect"

#### Severity
LOW

#### Scope
APP

#### Description
The spec and brief both specify the confirmation button text should be "Disconnect". The actual button text is "Confirm". The button has the correct `data-role="confirm-disconnect"` attribute and `phx-click="disconnect"` handler, but the visible label does not match the spec.

Source: `lib/metric_flow_web/live/integration_live/index.ex` — `<button ... class="btn btn-error">Confirm</button>`.

### Disconnect flash message missing "no new data will sync" clause

#### Severity
LOW

#### Scope
APP

#### Description
After confirming disconnect, the flash message reads: "Disconnected from {platform name}. Historical data is retained."

The spec requires: "Disconnected from {platform name}. Historical data is retained; no new data will sync after disconnecting."

The second clause is omitted. The modal body does include this text, but the post-disconnect flash does not.

Source: `lib/metric_flow_web/live/integration_live/index.ex` — `put_flash` in `handle_event("disconnect", ...)`.

### Connect button initiates OAuth redirect instead of showing informational flash

#### Severity
LOW

#### Scope
APP

#### Description
In the Available Platforms section, clicking "Connect" (`[data-role='reconnect-integration']`) immediately redirects the browser to the OAuth provider's authorization page. The brief expected a flash message "Reconnect {Platform}: authorize your account on the Connect page." to keep the user on the integrations page first.

The `handle_event("initiate_connect", ...)` handler calls `redirect(socket, to: ~p"/integrations/oauth/#{provider_str}")` which triggers immediate OAuth without any intermediate feedback to the user.

Source: `lib/metric_flow_web/live/integration_live/index.ex`, `handle_event("initiate_connect", ...)`.

Reproduced at: `http://localhost:4070/integrations`, click "Connect" on any available platform.

### Connected date format is YYYY-MM-DD instead of human-readable format

#### Severity
INFO

#### Scope
APP

#### Description
The `[data-role='integration-connected-date']` element displays dates as "Connected 2026-03-06" using `Calendar.strftime(integration.inserted_at, "%Y-%m-%d")`. The brief expected a human-readable format like "Connected Mar 05, 2026".

Source: `lib/metric_flow_web/live/integration_live/index.ex` — `Calendar.strftime(integration.inserted_at, "%Y-%m-%d")`.

### Stale google_ads integration record in database not cleared by seed script

#### Severity
LOW

#### Scope
QA

#### Description
The QA database had an existing `google_ads` integration record (provider = "google_ads") from a prior run, created before the seed was updated to use `:google`. The idempotent seed script checks for an existing `:google` integration and skips creation, but does not delete or migrate the stale `:google_ads` record. As a result, the Google integration was absent from the integrations page on initial load because `:google_ads` is not a registered provider.

The record was corrected manually via `UPDATE integrations SET provider = 'google' WHERE provider = 'google_ads'`. The seed script should be updated to either delete stale `:google_ads` records or update them to `:google` as part of its idempotent setup.

Source: `priv/repo/qa_seeds.exs` — missing cleanup for legacy `google_ads` provider records.
