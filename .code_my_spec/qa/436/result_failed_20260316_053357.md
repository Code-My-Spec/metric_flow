## Issues

### Disconnect confirmation uses inline panel instead of modal overlay

#### Severity
MEDIUM

#### Description
The spec (`index.spec.md`) describes the disconnect confirmation as a `.modal` overlay. The brief tests for `[class*='modal-open']`. The actual implementation renders an **inline collapsible panel** (`[data-role='disconnect-warning']`) inside the integration card rather than a DaisyUI modal.

No `[class*='modal-open']` element is ever present. The inline panel does work correctly (appears on disconnect click, dismissed by Cancel, executes disconnect on Confirm), but the UX differs from the spec design which calls for a modal overlay.

Reproduced at: `http://localhost:4070/integrations`, click "Disconnect" on any connected integration card.

### Confirm button in disconnect panel labeled "Confirm" not "Disconnect"

#### Severity
LOW

#### Description
The spec states the confirm button should have text "Disconnect" (with `data-role="confirm-disconnect"`). The actual button text is "Confirm". Both the spec and the brief explicitly expect "Disconnect" as the button label.

Selector: `[data-role='confirm-disconnect']`, actual text: "Confirm".

### Disconnect flash message missing "no new data will sync" clause

#### Severity
LOW

#### Description
After confirming disconnect, the flash message reads: "Disconnected from Google. Historical data is retained."

The spec (`index.spec.md`) and brief both expect the full message: "Disconnected from {platform name}. Historical data is retained; no new data will sync after disconnecting."

The second clause ("no new data will sync after disconnecting") is omitted in the actual flash. The disconnect warning panel body does include this text, but the post-disconnect flash does not.

Source: `lib/metric_flow_web/live/integration_live/index.ex`, line 321, `put_flash` in `handle_event("disconnect", ...)`.

### Connect button initiates OAuth instead of showing flash message

#### Severity
LOW

#### Description
The brief (Scenario 13) expects that clicking the "Connect" button in Available Platforms shows a flash message: "Reconnect Google Ads: authorize your account on the Connect page." Instead, clicking the button immediately redirects the browser to the provider's OAuth consent screen (e.g., `https://accounts.google.com/...`).

This may be the intended behavior (the `initiate_connect` event redirects to `/integrations/oauth/{provider}`), but it contradicts the brief's expected flash message behavior. The brief expectation may be incorrect based on how the feature is actually implemented.

Reproduced at: `http://localhost:4070/integrations`, click "Connect" on any available platform.

#### Scope
QA

### Seed script for `google_ads` integration fails — provider does not exist

#### Severity
MEDIUM

#### Scope
QA

#### Description
The brief's seed command creates a `google_ads` integration:

```
alias MetricFlow.Integrations.Integration
%Integration{} |> Integration.changeset(%{provider: :google_ads, ...})
```

However, the application's `list_providers/0` only returns `[:google, :facebook_ads, :quickbooks]`. The provider `:google_ads` is not registered in `@default_providers` in `lib/metric_flow/integrations.ex`. The seed command would insert an integration with an unrecognized provider key that would never appear in the integrations UI.

Additionally, the seed command uses `mix run -e` with inline Elixir, which fails when the Phoenix server is running due to the Cloudflare tunnel GenServer conflict (as documented in the QA plan).

The brief should reference `:google` (not `:google_ads`) and use a `.exs` script in `priv/repo/` instead of inline `mix run -e`.

### Integration sync status element empty at rest — "Connected" badge is a sibling

#### Severity
INFO

#### Description
The brief (Scenario 5) expects `[data-role='integration-sync-status']` to contain a `.badge-success` with "Connected" text. The actual implementation renders `[data-role='integration-sync-status']` as an empty span at rest. The "Connected" badge (`<span data-status="connected" class="badge badge-success">Connected</span>`) is a sibling element in the same flex row, not inside the sync status element.

The sync status element only shows content when: (a) a sync is actively running (shows a "Syncing" badge with spinner), or (b) after a sync completes (shows record count and timestamp).

The brief's selector for Scenario 5 is incorrect and should check for `[data-status='connected'].badge-success` instead of `[data-role='integration-sync-status'] .badge-success`.

#### Scope
QA
