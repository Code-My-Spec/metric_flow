## Issues

### "Welcome back!" flash not shown after password login

#### Severity
MEDIUM

#### Description
After a successful password login, `UserSessionController#create` sets a `put_flash(:info, "Welcome back!")` before calling `UserAuth.log_in_user/3`, which redirects to `/`. However, the home page (`/`) is rendered by `PageController#home` using `page_html/home.html.heex` as a template within `root.html.heex`. The root layout does not include `<.flash_group flash={@flash} />` — only `Layouts.app` (used by LiveViews) includes flash rendering.

As a result, the "Welcome back!" flash is set in the session but silently discarded — it is never rendered on the home page. Users have no confirmation that their login succeeded.

Reproduced: log in with `qa@example.com` / `hello world!` — the home page loads with no flash message.

Root cause: `root.html.heex` lacks `<.flash_group>`. Either the root layout needs to include flash rendering, or the post-login redirect should go to a LiveView page that uses `Layouts.app`.

### curl-based remember_me cookie tests blocked by CSRF protection

#### Severity
LOW

#### Scope
QA

#### Description
Scenarios 11 and 12 in the brief instruct using curl to POST `/users/log-in` and inspect the `_metric_flow_web_user_remember_me` cookie. However, all POST requests to the login endpoint without a valid CSRF token return `403 Forbidden`. Phoenix's CSRF plug rejects the request before the controller is invoked.

The brief should be updated to either use a browser-based cookie inspection approach (e.g., via JavaScript `document.cookie` for non-HttpOnly cookies, or a network trace) or acknowledge that the remember_me cookie check cannot be done via raw curl and provide an alternative verification method.
