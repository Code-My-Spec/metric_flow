## Issues

### White-label config cannot be updated via UI once saved — phx-change overwrites user edits

#### Severity
HIGH

#### Description
When a `WhiteLabelConfig` already exists for the account, editing form fields in `#white-label-form` triggers `phx-change="validate_white_label"`. The handler updates `white_label_form` assigns, but the template's `white_label_value/3` helper always reads from `@white_label_config` (the saved struct) when it is non-nil, completely ignoring the form params. This causes the LiveView to re-render the input `value` attributes from the saved config, silently overwriting whatever the user typed.

The result: when submitting the form, only the value in the last-filled field (the one filled most recently before submit, before any subsequent phx-change re-render can reset it) reflects the user's new input. All other fields revert to the saved config values in the DOM and are submitted with their old values.

**Reproduction:**
1. Save any white-label config (subdomain, logo URL, colors)
2. Return to `/accounts/settings`
3. Type a new value into the logo URL field — the field visually shows the new value momentarily, then reverts to the saved value after phx-change fires
4. Submit the form — the logo URL is submitted with the old saved value, not the new value

**Root cause:** `white_label_value(%WhiteLabelConfig{} = config, _form, field)` ignores `_form` entirely. The `validate_white_label` handler should merge form params into the existing config for display purposes, or `white_label_value` should prefer form params when they are non-empty.

**File:** `lib/metric_flow_web/live/agency_live/settings.ex` — `white_label_value/3` (line 302–306); `lib/metric_flow_web/live/account_live/settings.ex` — `handle_event("validate_white_label", ...)` (line 473)
