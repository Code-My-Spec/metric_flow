# QA Plan

## App Overview

MetricFlow is a Phoenix LiveView application for marketing analytics and reporting. Authentication uses Phoenix's session-based approach generated by `phx.gen.auth` with a magic-link primary flow and optional password login — the `UserSessionController` handles the `POST /users/log-in` form submission, sets a signed session cookie, and redirects on success. The app runs on port 4070 in development (`http://localhost:4070`). Routes are organized into two main LiveView sessions: `require_authenticated_user` (guarded by `require_authenticated_user` plug + `UserAuth.require_authenticated` on_mount hook) covering all feature pages, and `current_user` (unguarded) covering registration, login, and onboarding.

## Tools Registry

### vibium MCP server (browser automation)

Use for all LiveView pages, form interactions, visual verification, and screenshot evidence. The vibium MCP server drives a real browser instance and handles LiveView's WebSocket upgrade automatically. All vibium tools are called directly as MCP tool calls — no CLI daemon needed.

Session setup — launch a browser once at the start of QA:

```
mcp__vibium__browser_launch(headless: true)
```

Auth workflow — log in via the password form:

```
mcp__vibium__browser_navigate(url: "http://localhost:4070/users/log-in")
mcp__vibium__browser_scroll_into_view(selector: "#login_form_password")
mcp__vibium__browser_fill(selector: "#login_form_password_email", text: "qa@example.com")
mcp__vibium__browser_fill(selector: "#user_password", text: "hello world!")
mcp__vibium__browser_click(selector: "#login_form_password button[name='user[remember_me]']")
mcp__vibium__browser_wait(selector: "body", timeout: 5000)
```

Navigate to authenticated pages after login:

```
mcp__vibium__browser_navigate(url: "http://localhost:4070/accounts")
mcp__vibium__browser_screenshot(filename: "screenshot.png")
```

Key vibium MCP tool patterns:

```
# Read page content
mcp__vibium__browser_get_text()                           # all visible text on current page
mcp__vibium__browser_get_text(selector: "h1")             # text of a specific selector
mcp__vibium__browser_get_html()                           # raw HTML of current page

# Find elements
mcp__vibium__browser_find(selector: "selector")           # find one element, returns tag/text/box
mcp__vibium__browser_find(role: "button", text: "Submit") # find by ARIA role + text
mcp__vibium__browser_find_all(selector: "selector")       # find all matching elements

# Interact
mcp__vibium__browser_fill(selector: "selector", text: "text")    # clear + type into an input
mcp__vibium__browser_type(selector: "selector", text: "text")    # type without clearing first
mcp__vibium__browser_click(selector: "selector")                  # click an element
mcp__vibium__browser_select(selector: "selector", value: "val")  # select a dropdown option

# Wait for LiveView updates
mcp__vibium__browser_wait(selector: "selector")                          # wait for element (attached)
mcp__vibium__browser_wait(selector: "selector", state: "visible")        # wait for element to be visible
mcp__vibium__browser_wait(selector: "selector", state: "hidden")         # wait for element to disappear
mcp__vibium__browser_wait_for_url(pattern: "/accounts")                  # wait for URL to contain string
mcp__vibium__browser_wait_for_load()                                     # wait for page load complete

# URL and state
mcp__vibium__browser_get_url()                            # get current URL
mcp__vibium__browser_get_title()                          # get current page title
mcp__vibium__browser_evaluate(expression: "JS code")      # run arbitrary JavaScript

# Screenshot
mcp__vibium__browser_screenshot(filename: "evidence.png")                # viewport screenshot
mcp__vibium__browser_screenshot(filename: "full.png", fullPage: true)    # full-page screenshot
mcp__vibium__browser_screenshot(annotate: true)                          # annotate interactive elements

# Scroll
mcp__vibium__browser_scroll_into_view(selector: "selector")  # scroll element into view

# Session management
mcp__vibium__browser_delete_cookies()                     # clear all cookies (for logout)
mcp__vibium__browser_quit()                               # close the browser session
```

### curl (unauthenticated HTTP checks)

Use for checking HTTP status codes and verifying redirects on unauthenticated access. For authenticated testing, use vibium MCP tools.

```bash
# Unauthenticated check — should redirect
curl -s -o /dev/null -w "%{http_code}" http://localhost:4070/accounts
# Expected: 302
```

### mix run (seed data)

Use to set up test data before running tests. All seed scripts are idempotent — safe to run multiple times.

```bash
# Main QA seeds — creates two users and a team account
mix run priv/repo/qa_seeds.exs
```

### start-qa.sh (seed data entry point)

Run this once at the start of each QA session to seed data. After running, use vibium MCP tools to launch the browser and log in.

```bash
./.code_my_spec/qa/scripts/start-qa.sh
```

Then launch browser and log in via MCP:

```
mcp__vibium__browser_launch(headless: true)
```

## Seed Strategy

### priv/repo/qa_seeds.exs

Run with: `mix run priv/repo/qa_seeds.exs`

Creates:

- **QA owner user**: `qa@example.com` / `hello world!`
  - Email is confirmed via magic link token so password login works
  - Suitable for testing all authenticated routes as an account owner
- **QA member user**: `qa-member@example.com` / `hello world!`
  - Second confirmed user for multi-user scenarios (invite, role changes, etc.)
- **Team account**: "QA Test Account"
  - Type: `team`, qa@example.com is the owner

Credentials printed at the end of the script:

```
Owner:    qa@example.com / hello world!
Member:   qa-member@example.com / hello world!
URL:      http://localhost:4070/users/log-in
Dev mailbox: http://localhost:4070/dev/mailbox
```

Run this script before any story that requires a logged-in user. Registration stories that test the form submission flow should use a fresh unique email rather than the seeded user, to avoid duplicate-email conflicts.

### Switching users

To switch between users during a QA session, clear cookies and re-login via MCP:

```
# Log out by clearing cookies
mcp__vibium__browser_delete_cookies()

# Navigate to login and authenticate as a different user
mcp__vibium__browser_navigate(url: "http://localhost:4070/users/log-in")
mcp__vibium__browser_scroll_into_view(selector: "#login_form_password")
mcp__vibium__browser_fill(selector: "#login_form_password_email", text: "qa-member@example.com")
mcp__vibium__browser_fill(selector: "#user_password", text: "hello world!")
mcp__vibium__browser_click(selector: "#login_form_password button[name='user[remember_me]']")
mcp__vibium__browser_wait(selector: "body", timeout: 5000)
```

## Notes

Before running browser-based QA tests, restart the Phoenix server (`mix phx.server`) to ensure the running BEAM has the latest compiled modules. Hot reload does not always propagate module attribute changes — a full server restart is required after significant source code changes. After restarting the server, quit and relaunch the browser (`browser_quit` then `browser_launch`) to get a fresh LiveView WebSocket session — stale sessions will silently drop form submissions.

The login page (`/users/log-in`) renders two forms with different IDs: `#login_form_magic` (email-only, sends a magic link) and `#login_form_password` (email + password, submits via `POST /users/log-in`). Always scroll the password form into view before interacting, since it may be below the fold.

After registration, the app shows a confirmation screen (not an immediate redirect) and sends a magic link email. The user must click the link to confirm before they can log in with a password. In QA testing, the dev mailbox at `/dev/mailbox` shows sent emails. The seed script bypasses this by directly calling `login_user_by_magic_link/1` to confirm the user programmatically.

The `/onboarding` route is in the public `current_user` live session (no auth required). After login, the server redirects to `/onboarding` if that is the `signed_in_path`. When testing post-login redirects, use `mcp__vibium__browser_wait_for_load()` or `mcp__vibium__browser_wait(selector: "body")` after clicking submit.

`account_name` is a required field on the registration form. The `registration_changeset` validates `account_name` as required. Submitting registration without an account name will fail validation. The `account_type` field is optional — the form shows a select with a "Select account type" prompt as the default.

Password minimum length is 12 characters. The seed password `"hello world!"` is exactly 12 characters and passes validation.

Use `mcp__vibium__browser_fill` (not `browser_type`) when filling form fields, as it clears existing content first. Use `browser_type` only when you need to append to existing content.

To check if an element has a specific attribute (e.g., readonly), use `mcp__vibium__browser_evaluate(expression: "document.querySelector('#selector')?.hasAttribute('readonly') ?? false")`.
