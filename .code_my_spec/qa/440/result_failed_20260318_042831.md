### Scenario 3: Sync Now on expired integration shows reconnection message

pass (code review finding — bug previously reported has been fixed)

The prior QA run (dated 2026-03-18) reported a HIGH bug: when `DataSync.sync_integration/2` returned `{:error, :not_connected}`, the LiveView showed `"Integration not found."` instead of a reconnection message.

**This bug has been fixed.** Current code in `lib/metric_flow_web/live/integration_live/index.ex` (lines 300–301) now handles `:not_connected` separately:

```elixir
{:error, :not_connected} ->
  {:noreply, put_flash(socket, :error, "Your #{platform_name} token has expired. Please reconnect.")}
```

The message format matches the acceptance criterion. The data flow is:
1. User clicks "Sync Now" on a Google Analytics card
2. LiveView calls `DataSync.sync_integration(scope, :google_analytics)`
3. `DataSync.check_connected/1` sees `expired_at` is in the past and `refresh_token` is present (QA seed sets `"qa_expired_refresh_token"`)
4. Wait — if a refresh token is present, `check_connected/1` returns `:ok` and the Oban job is dispatched

**Important nuance identified:** The QA seed creates the integration with `refresh_token: "qa_expired_refresh_token"`. The `check_connected/1` function (`lib/metric_flow/data_sync.ex` lines 155–159) returns `{:error, :not_connected}` only when the token is expired AND **no refresh token is available**. With a refresh token present, the integration passes the check and an Oban job is dispatched — the worker then attempts a refresh, fails with `{:error, :token_expired}`, and records failure history. The flash message shown would be `"Sync started for Google Analytics"`, not the expired token reconnection message.

To trigger the `:not_connected` path and see the reconnection flash, the seed would need to create the integration without a `refresh_token`. This is a gap in the seed script.

Visual confirmation of the flash message could not be performed.

partial (code review)

For the QA seed case (which DOES have a refresh token), the Oban job would be dispatched, attempt a refresh against a fake token URL, likely fail with a network error rather than `{:error, :token_expired}`, and would not send an email.

## Issues

### Seed script creates integration with refresh token, preventing :not_connected error path

#### Severity
MEDIUM

#### Scope
QA

#### Description
The QA seed script `priv/repo/qa_seeds_440.exs` creates the expired `google_analytics` integration with `refresh_token: "qa_expired_refresh_token"`. However, `DataSync.check_connected/1` only returns `{:error, :not_connected}` when the token is expired AND no refresh token is available:

```elixir
defp check_connected(%Integration{} = integration) do
  if Integration.expired?(integration) and not Integration.has_refresh_token?(integration) do
    {:error, :not_connected}
  else
    :ok
  end
end
```

With a refresh token present, `check_connected/1` returns `:ok`, the Oban job is dispatched, and the flash shown to the user is `"Sync started for Google Analytics"` — not the reconnection message. Scenario 3 of the brief (testing that "Sync Now" shows a reconnection flash) cannot be exercised with the current seed.

To test the `:not_connected` path, the seed script should create the integration without a `refresh_token` (or with `refresh_token: nil`). Update `priv/repo/qa_seeds_440.exs` to remove the `refresh_token` field or set it to `nil`.

### Email notification not sent on "Sync Now" — only on scheduled sync

#### Severity
LOW

#### Description
The acceptance criterion "user receives email notification when credentials expire" is only partially met. Email notification is implemented in `MetricFlow.DataSync.SyncWorker` and fires when a scheduled Oban sync job encounters a token expiry error during refresh. It is NOT triggered by the "Sync Now" UI button path because `DataSync.check_connected/1` blocks job dispatch before the worker runs.

Practical impact: a user whose token expires would only receive an email after the next daily scheduled sync (02:00 UTC) attempts and fails to refresh their token — not immediately when they try to sync manually. If the user never has a refresh token (OAuth scopes did not include offline access), no email is ever sent.

This may be by design, but it is worth documenting for product review.
