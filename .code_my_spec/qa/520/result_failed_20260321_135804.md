**Result: FAIL**

The page pre-filled the manual input with `accounts/123/locations/deleted-loc-1` but showed no warning, flag, or indicator. No `[data-role='missing-location']` element was visible. No text containing "missing", "unavailable", "removed", or "revoked" was present.

The template already renders `[data-role='missing-location']` and `[data-role='location-unavailable']` (lines 292–304 of `connect.ex`), but it can only fire when the live API returns location data. This scenario cannot be fully validated without real GBP API access.

## Issues

### Missing-location flagging requires live GBP API (criterion 4862 not fully testable in QA)

#### Severity
HIGH

#### Description
When a previously configured location ID is no longer available (e.g., a deleted or revoked location), the accounts page shows no warning. The manual entry field is pre-filled with the stale location ID with no UI feedback.

The `compute_missing_locations/3` function (`connect.ex` lines 748–757) only computes missing locations when `fetched_accounts != []`. Since the live GBP API requires valid OAuth credentials, a test token produces no API results (`list_google_business_locations/1` returns empty or errors), so the comparison never runs and the warning never appears.

The template already has the full UI for this feature — `[data-role='missing-location']` and `[data-role='location-unavailable']` are rendered (lines 292–304 of `connect.ex`) — but the logic gate `fetched_accounts != []` prevents the warning from ever showing in QA.

Reproduced by: setting `included_locations` to `["accounts/123/locations/deleted-loc-1"]` in the integration's `provider_metadata` and navigating to `/integrations/connect/google_business/accounts` with a test (non-real) OAuth token.
