## Issues

### Sync Now button never enters disabled state after click

#### Severity
MEDIUM

#### Description
After clicking "Sync Now" on a connected integration card, the button immediately fires the `phx-click="sync"` event. The server adds the platform key to `@syncing` and responds with the "Sync started for ..." flash. The sync worker then runs and completes (or fails) asynchronously — in dev, this happens so fast that the LiveView diff containing the disabled button is never painted before the completion event clears `@syncing`. The result is that the button is never visually disabled from the user's perspective.

Reproduced at `http://localhost:4070/integrations` with the Facebook Ads connected integration. Click "Sync Now" and observe that the button remains enabled throughout. The "Sync started for..." and "Sync failed for..." flashes appear nearly simultaneously.

This may also affect real environments when sync workers are fast (cached data, small datasets). The `@min_syncing_ms = 1_000` guard defers `sync_completed` and `sync_failed` PubSub messages to ensure at least 1 second of loading state — but this deferral requires that the initial `@syncing` MapSet assignment renders first. If the PubSub message arrives before the LiveView sends the first diff to the client, the deferred path is not taken correctly.

### Syncing badge loading spinner never renders

#### Severity
MEDIUM

#### Description
The "Syncing" badge with `badge-warning` class and embedded `<span class="loading loading-spinner loading-xs ml-1">` is expected to appear in `[data-role='integration-sync-status']` while a sync is in progress. This badge was never observed during testing. The `@min_syncing_ms` 1-second minimum display guard exists in the code, but the guard relies on the PubSub `{:sync_completed, ...}` or `{:sync_failed, ...}` messages arriving after the `@syncing` MapSet is set. If the worker completes before the client WebSocket receives the first diff, the loading state is invisible.

Reproduced at `http://localhost:4070/integrations` — click "Sync Now" and observe no Syncing badge ever appears in the connected card's status area.

### Seed data drift — Facebook Ads connected instead of Google

#### Severity
LOW

#### Scope
QA

#### Description
The base `qa_seeds.exs` script creates a Google integration for `qa@example.com`, but the integrations page showed Facebook Ads as the connected platform. Previous story 436 (View and Manage Platform Integrations) test runs disconnected the Google integration and a Facebook Ads integration was inserted, leaving the database in a different state than the seeds expect.

The `qa_seeds.exs` script is not fully idempotent for integrations — it creates a Google integration if none exists, but does not remove stale integrations from other providers. For story 438, the Sync Now functionality was testable via the Facebook Ads card, so this did not block testing. However, the brief's seed notes state "qa@example.com has a connected Google integration" which was not accurate.

To fix: Add a cleanup step to `qa_seeds.exs` that removes any non-Google integrations for `qa@example.com` before creating the expected Google integration, or add a `qa_seed_438.exs` that establishes known integration state.
