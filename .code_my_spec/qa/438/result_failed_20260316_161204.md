### Scenario 3 — Clicking Sync Now triggers sync and shows loading state

partial

Clicked `button[phx-click="sync"]`. The flash message "Sync started for Google" appeared in the page text, confirming the event was handled and `DataSync.sync_integration/2` returned `{:ok, _sync_job}`.

However, the "Syncing" badge with spinner inside `[data-role="integration-sync-status"]` was never rendered when the screenshot was captured. The Oban worker completes and broadcasts `{:sync_failed, ...}` before the LiveView WebSocket delivers the DOM diff showing the loading state to the browser. The `@syncing` MapSet was populated then immediately cleared within the same render cycle visible to the browser automation tool.

The button was re-enabled by the time the screenshot was taken. Criterion 4051/4052 (loading state) requires `mix spex` unit tests to validate, as the BDD spex directly sends `{:sync_completed, ...}` / `{:sync_failed, ...}` messages to the LiveView PID.

Evidence: `.code_my_spec/qa/438/screenshots/s3-after-sync-click.png`, `.code_my_spec/qa/438/screenshots/s3-sync-flash-messages.png`

## Issues

### Sync Now button never triggers actual data sync — Oban job fails with stub token in dev

#### Severity
MEDIUM

#### Scope
QA

#### Description
Clicking "Sync Now" in the dev environment with the seeded integration (which uses `access_token: "qa_test_access_token"`) triggers an Oban job that immediately fails when the Google data provider attempts to use the stub token against the real Google API. The result is a "Sync failed" flash with no data synced.

This means Scenario 4 (success state after sync completes) cannot be verified via browser automation with the current seed data. The `[data-role="integration-sync-status"]` badge showing "Synced N records at YYYY-MM-DD" is untestable end-to-end without either a real OAuth token or a dev-mode stub that returns fake metrics.

Repro: Log in as `qa@example.com`, navigate to `/integrations`, click "Sync Now" on the Google card. Flash will show "Sync started for Google" then immediately "Sync failed for Google. Please check your connection and try again."

Consider adding a dev-environment stub to `DataSync.sync_integration/2` or the Google data provider that returns a canned success response when the access token matches a known test value (e.g. `"qa_test_access_token"`).

### Syncing loading state (badge and spinner) not observable via browser automation

#### Severity
LOW

#### Scope
QA

#### Description
The "Syncing" badge with spinner inside `[data-role="integration-sync-status"]` is rendered only while the provider is in the `@syncing` MapSet. In the dev environment, the Oban sync job completes (and broadcasts `{:sync_completed, ...}` or `{:sync_failed, ...}`) before the LiveView WebSocket delivers the DOM diff showing the loading state to the browser.

The brief (Scenario 3, steps 6-7) asks the QA agent to assert the "Syncing" badge is visible and the spinner is present immediately after clicking. This assertion cannot pass when the Oban job completes within the same LiveView render window.

This is an inherent timing issue with in-process Oban in dev. It is best validated through the BDD spex (criterion 4051/4052) which send the async messages manually to the LiveView PID via `send(view.pid, {:sync_completed, ...})`.
