The "Syncing" badge with spinner (`[data-role="integration-sync-status"]` containing a `.badge.badge-warning`) was not observable via browser automation. The "Sync started for Google" flash confirms criterion 4051 (flash on trigger) passes. The loading badge (criteria 4051/4052) requires the BDD spex unit tests (`mix spex`) to validate, as those tests send `{:sync_completed, ...}` / `{:sync_failed, ...}` messages directly to the LiveView PID.

## Issues

### Sync Now button never triggers actual data sync — Oban job fails with stub token in dev

#### Severity
MEDIUM

#### Scope
QA

#### Description
Clicking "Sync Now" in the dev environment with the seeded integration (which uses `access_token: "qa_test_access_token"`) triggers an Oban job that immediately fails when the Google data provider attempts to use the stub token against the real Google API. The result is a "Sync failed" flash with no data synced.

This means Scenario 4 (success state after sync completes) cannot be verified via browser automation with the current seed data. The `[data-role="integration-sync-status"]` badge showing "Synced N records at YYYY-MM-DD" is untestable end-to-end without either a real OAuth token or a dev-mode stub that returns fake metrics.

Repro: Log in as `qa@example.com`, navigate to `/integrations`, click "Sync Now" on the Google card. Flash shows "Sync started for Google" then immediately "Sync failed for Google. Please check your connection and try again."

Consider adding a dev-environment stub to the Google data provider that returns a canned success response when the access token matches a known test value such as `"qa_test_access_token"`.

### Syncing loading state (badge and spinner) not observable via browser automation

#### Severity
LOW

#### Scope
QA

#### Description
The "Syncing" badge with spinner inside `[data-role="integration-sync-status"]` is rendered only while the provider is in the `@syncing` MapSet. In the dev environment, the Oban sync job completes (broadcasting `{:sync_failed, ...}`) before the LiveView WebSocket delivers the DOM diff for the loading state to the browser.

Scenario 3, steps 6-7 in the brief ask the QA agent to assert the "Syncing" badge and spinner are visible immediately after clicking. This assertion cannot pass when the Oban job completes within the same LiveView render window.

This is an inherent timing constraint with in-process Oban in dev. The loading state is best validated through the BDD spex (criteria 4051/4052) which send the async messages manually to the LiveView PID via `send(view.pid, {:sync_completed, ...})`.
