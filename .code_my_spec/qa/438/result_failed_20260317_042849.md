### Scenario 5 — Button is disabled after clicking Sync Now

FAIL

After clicking Sync Now, the button did NOT receive a `disabled` attribute. The DOM showed the button with no `disabled` attribute and `browser_is_enabled` returned `true`. This means the "syncing" state was either never applied to the LiveView assigns, or the sync completed/failed before the LiveView re-render reached the client.

The sync worker completes (fails) so quickly in dev that the `@syncing` MapSet is populated by `handle_event("sync", ...)` and then immediately cleared by `handle_info({:sync_failed, ...}, ...)` before the client renders the intermediate state. The `phx-disable-with="Please wait..."` attribute on the button is also not sufficient because that only disables the button during the phx event round-trip — it re-enables after the server sends a response, not after the background job completes.

Expected: `[data-platform="facebook_ads"] button[phx-click="sync"]` should have `disabled` attribute while syncing.
Actual: No `disabled` attribute present.

FAIL

The sync worker completes/fails before the LiveView's PubSub handler can set `@syncing` in the assigns and push a re-render that reaches the client. The Syncing badge code path (`MapSet.member?(@syncing, platform.key)`) is never observed in the rendered HTML.

## Issues

### Sync Now button never enters disabled/in-progress state

#### Severity
HIGH

#### Description
After clicking "Sync Now", the button does not become disabled and no "Syncing" badge appears in the `[data-role="integration-sync-status"]` span. The criteria (4051, 4052) require a visible loading state while sync is in progress.

The root cause is a race condition: `handle_event("sync", ...)` assigns `platform_key` to `@syncing` and returns `{:noreply, socket}`, which should push a re-render to the client. However, the sync worker runs synchronously enough in the dev environment that `handle_info({:sync_failed, ...})` fires and removes the key from `@syncing` before the intermediate state is painted in the browser.

Reproduced by clicking "Sync Now" on the Facebook Ads card at `http://localhost:4070/integrations`. In the rendered HTML immediately after the click, `button[phx-click="sync"]` has no `disabled` attribute and `.badge-warning` is absent from the DOM.

The `phx-disable-with` attribute (`"Please wait..."`) only disables the button for the duration of the LiveView event round-trip (until the server sends a diff), not for the duration of the background job. Once `handle_event` returns, LiveView re-enables the button even though the sync worker has not finished.

To fix: the disabled state must be driven by `@syncing` in the template, not by `phx-disable-with`. The template already has `disabled={MapSet.member?(@syncing, platform.key)}` — but the button is not rendered as disabled even immediately after the click. This suggests either the sync worker completes before the client processes the first re-render, or the PubSub `sync_failed` message is received in the same LiveView process tick that handled the event, causing both assigns to cancel each other before the first render diff is sent.

URL: `http://localhost:4070/integrations`

### Sync failure flash message is generic rather than actionable

#### Severity
LOW

#### Description
When the sync fails (as it always does in dev with a `qa_test_token`), the error flash reads: "Sync failed for Facebook: All data providers failed. Check your integration settings." The phrase "All data providers failed" is not user-friendly and does not explain what to check.

This is a dev-only behavior since real tokens are not configured, but the message text is generated from `build_failure_message/2` in `index.ex` and should provide clearer guidance.

URL: `http://localhost:4070/integrations`

### Seed data drift — Facebook instead of Google integration present

#### Severity
LOW

#### Scope
QA

#### Description
The brief states the expected seed state as "qa@example.com has a connected Google integration (provider: :google)" and expects Google Analytics and Google Ads to appear in the Connected Platforms section. However, the actual seed state has a Facebook Ads integration (provider: :facebook_ads) as the only connected platform. Google Analytics, Google Ads, and QuickBooks are all in the Available section.

This caused the brief's platform-specific selectors (e.g., `[data-platform='google_analytics'] button[phx-click='sync'][disabled]`) to be inapplicable. Testing was adapted to use Facebook Ads instead.

The QA seeds script (`priv/repo/qa_seeds.exs`) and/or the seed state for story 438 should be audited to ensure the expected platforms are created.
