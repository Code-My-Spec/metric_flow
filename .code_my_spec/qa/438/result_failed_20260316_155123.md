### Scenario 3 — Clicking Sync Now triggers sync and shows loading state

partial

Clicked `button[phx-click="sync"]`. The flash message "Sync started for Google" appeared, confirming the event was handled and `DataSync.sync_integration/2` returned `{:ok, _sync_job}`.

However, the "Syncing" badge with spinner (`[data-role="integration-sync-status"]` containing a `.badge-warning` with "Syncing" text) never rendered. The Oban worker completed and broadcast `{:sync_failed, ...}` before the LiveView had a chance to re-render the loading state. The `@syncing` MapSet was populated then immediately cleared in the same render cycle visible to the browser.

This means the syncing loading state (criterion 4052) is not observable via browser automation when the Oban job completes synchronously in dev. The button was also immediately re-enabled and the sync-status span remained empty.

Evidence: `.code_my_spec/qa/438/screenshots/03_after_sync_click.png`, `.code_my_spec/qa/438/screenshots/04_sync_started_and_failed_flash.png`

### Scenario 4 — Success state after sync completes

This failure is expected in the dev environment with a stub token — it is not a regression in the Sync Now trigger flow. The sync flow through criterion 4053 (success state rendering) was verified separately via `mix spex` unit tests.

## Issues

### Sync Now button never triggers actual data sync — Oban job fails with stub token in dev

#### Severity
MEDIUM

#### Description
Clicking "Sync Now" in the dev environment with the seeded integration (which uses `access_token: "qa_test_access_token"`) triggers an Oban job that immediately fails when the Google data provider attempts to use the stub token against the real Google API. The result is a "Sync failed" flash with no data synced.

This means Scenario 4 (success state) cannot be verified via browser automation with the current seed data. The `[data-role="integration-sync-status"]` badge showing "Synced N records at YYYY-MM-DD" is untestable end-to-end without either a real OAuth token or a dev-mode stub that returns fake metrics.

Repro: Log in as `qa@example.com`, navigate to `/integrations`, click "Sync Now" on the Google card. Flash will show "Sync started for Google" then immediately "Sync failed for Google. Please check your connection and try again."

Consider adding a dev-environment stub to `DataSync.sync_integration/2` or the Google data provider that returns a canned success response when the access token matches a known test value (e.g. `"qa_test_access_token"`).

#### Scope
QA

### Syncing loading state (badge + spinner) not observable via browser automation

#### Severity
LOW

#### Description
The "Syncing" badge with spinner inside `[data-role="integration-sync-status"]` is rendered only while the provider is in the `@syncing` MapSet. In the dev environment, the Oban sync job completes (and broadcasts `{:sync_completed, ...}` or `{:sync_failed, ...}`) before the LiveView WebSocket delivers the DOM diff showing the loading state to the browser.

The brief (Scenario 3, step 6–7) asks the QA agent to assert the "Syncing" badge is visible and the spinner is present immediately after clicking. This assertion cannot pass when the Oban job completes synchronously within the same LiveView render window.

This is an inherent timing issue with in-process Oban in dev. It is best validated through the BDD spex (criterion 4051/4052) which send the async messages manually to the LiveView PID.

#### Scope
QA
