## Issues

### Sync Now always fails with "all_providers_failed" in QA environment — sync success path untestable

#### Severity
MEDIUM

#### Scope
QA

#### Description

The QA seed script creates a Google integration with a stub access token (`qa_test_access_token`). When Sync Now is clicked, the Oban SyncWorker runs immediately and attempts real API calls to Google Analytics and Google Ads. Both fail because the stub token is not a valid OAuth bearer token. The `run_all_providers` path returns `{:error, :all_providers_failed}` and broadcasts `{:sync_failed, ...}` to the LiveView, which shows the error flash "Sync failed for google: all_providers_failed".

As a result, Scenario 4 (success state after sync completes) and the intermediate loading state in Scenario 3 cannot be verified via browser automation. The success flash, sync result badge, and post-completion button re-enable state all depend on a successful sync response from the data providers.

To make these scenarios testable in QA, one of the following approaches is needed:
- Seed the database with a real (non-expired) OAuth token for a test account
- Add a dev-mode stub data provider that returns fake metrics without making real API calls
- Use `mix spex` to run the BDD spex (criteria 4053, 4054) which simulate the async messages directly via `send(context.view.pid, {:sync_completed, ...})` — this is explicitly noted in the brief as the recommended path for those criteria

The current failing behavior also produces a misleading UX: the user sees both "Sync started for Google" (info) and "Sync failed for google: all_providers_failed" (error) simultaneously. The error message "all_providers_failed" is a raw atom string — not user-friendly. This is a separate app-level concern noted below.

### "Sync failed for google: all_providers_failed" error message is not user-friendly

#### Severity
LOW

#### Scope
APP

#### Description

When a sync attempt fails because all data providers return errors, the LiveView broadcasts `{:sync_failed, %{provider: :google, reason: :all_providers_failed}}`. The `handle_info/2` handler renders this as the error flash: "Sync failed for google: all_providers_failed".

Two issues:
1. The provider name is rendered as the raw atom string `"google"` rather than the display name "Google". The `format_error/1` helper in `SyncWorker` uses `Atom.to_string/1`, and the `handle_info` clause in `index.ex` uses `provider` directly rather than looking up the display name.
2. "all_providers_failed" is an internal error atom, not a user-facing message. Users would not understand what this means. A clearer message would be: "Sync failed for Google. Please check your connection and try again."

Reproduced by: logging in as `qa@example.com`, navigating to `http://localhost:4070/integrations`, clicking "Sync Now" on the Google card.
