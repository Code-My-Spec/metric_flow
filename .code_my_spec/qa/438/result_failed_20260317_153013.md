## Issues

### Sync Now disabled state not observable in dev environment

#### Severity
LOW

#### Scope
APP

#### Description
After clicking "Sync Now", the `disabled` attribute on `button[phx-click='sync']` was never observable. The template correctly sets `disabled={MapSet.member?(@syncing, platform.key)}` (index.ex line 134), which should disable the button while the sync is in progress. However, in the dev environment, the sync worker completes and sends the `{:sync_completed, ...}` PubSub message almost immediately â€” the LiveView processes the completion and re-renders (removing the disabled state and the Syncing badge) before a browser check can observe the intermediate state.

This means the "loading indicator" and "button disabled while syncing" behaviors described in criteria 4051 and 4052 cannot be verified end-to-end in a dev environment with a fast sync worker. The code paths for these states are present in the template and event handlers, but a real slow API would be needed to observe them, or the sync worker would need an artificial delay.

The flash message ("Sync started for Google Analytics") and the final sync result badge ("Synced 186 records") both work correctly, confirming the event flow is wired end-to-end.

### Syncing badge and loading spinner not observable in dev

#### Severity
LOW

#### Scope
APP

#### Description
The "Syncing" badge (`badge-warning` with `.loading-spinner`) defined at index.ex lines 96-99 is never visible in the dev environment. The sync job completes before the LiveView re-render for the "syncing" state can be observed â€” the card transitions directly from "no badge" to "Synced N records" badge without showing the intermediate "Syncing" state.

This is not a code bug â€” the template logic is correct. It is an observability gap: the sync worker in dev runs against a cassette or mock that returns instantly, so the `@syncing` MapSet never persists long enough to render to the browser.

To test this behavior, either a real slow API integration or an artificial delay in the sync worker would be needed. Alternatively, a LiveView unit test that mocks the PubSub timing would provide coverage.
