## Issues

### Sync Now button never enters disabled/loading state in practice

#### Severity
MEDIUM

#### Description
When a user clicks "Sync Now", the button has `disabled={MapSet.member?(@syncing, platform.key)}`
in the template and `phx-disable-with="Please wait..."`. The LiveView event handler adds the
platform key to `@syncing`, which should cause a re-render with the button disabled and the
"Syncing" badge showing.

In practice, the sync worker completes (or fails) so quickly — and the resulting
`{:sync_failed, ...}` PubSub message clears `@syncing` — that the intermediate disabled state
never visibly renders in the browser. Users clicking Sync Now see no visual confirmation that
the sync is in progress. They receive a "Sync started for..." flash, then a "Sync failed for..."
flash in rapid succession, with no loading indicator between them.

Reproduced consistently on `http://localhost:4070/integrations` with the Google integration.

### Sync failure flash message exposes raw Elixir tuple for Google sync exceptions

#### Severity
HIGH

#### Description
When the Google sync worker catches an exception (e.g., a Postgres string_data_right_truncation
error), it returns `{:error, {:exception, "ERROR 22001 ..."}}`. The `format_error/1` function
in `SyncWorker` does not handle the `{:exception, reason}` tuple shape, so it falls through to
the catch-all `inspect(reason)` clause, producing the flash message:

```
Sync failed for Google: {:exception, "ERROR 22001 (string_data_right_truncation) value too long for type character varying(255)"}
```

This exposes internal Elixir error structure and a raw Postgres error string to the user. The
`format_error` function should handle the `{:exception, message}` tuple shape and return a
friendly message such as "An unexpected error occurred during sync. Please try again."

The underlying cause appears to be the Google sync worker attempting to write data with a field
that exceeds the database column length (255 chars), likely a metadata field or account identifier
that is longer than the schema allows.

Reproduced at `http://localhost:4070/integrations` by clicking Sync Now on Google Analytics or
Google Ads when a Google integration is connected.

### Google sync fails with database truncation error (string_data_right_truncation)

#### Severity
HIGH

#### Description
The Google sync worker throws an Ecto exception with `ERROR 22001 (string_data_right_truncation)
value too long for type character varying(255)`. The sync is failing at the database write layer,
not at the API layer. This suggests a database schema column (likely in the sync job or metric
records) is too short to hold the data being written from the Google API response.

The sync completes its API fetch phase but fails when persisting results, meaning every manual
sync attempt for Google integrations will fail silently (from the user's perspective, they just
get an error flash with no records synced).

Reproduced at `http://localhost:4070/integrations` by clicking Sync Now on any Google platform
(Google Analytics or Google Ads) when a Google integration with `access_token: "qa_test_token"`
is present.

### Facebook sync fails with "All data providers failed"

#### Severity
LOW

#### Description
Clicking Sync Now on the Facebook Ads card produces: "Sync failed for Facebook: All data providers
failed. Check your integration settings." This is expected in dev (no real Facebook access token),
but is noted as a finding. The error message is user-friendly (does not expose internal error
details), which is correct behavior.

This is expected in the dev/test environment and is not a production bug, but is documented as
evidence that the sync failure handling path works for Facebook whereas it is broken for Google.
