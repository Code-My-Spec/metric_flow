## Issues

### Role change via phx-change select does not update role or show flash

#### Severity
HIGH

#### Scope
APP

#### Description
On `/accounts/members`, changing a member's role using the inline `<select phx-change="change_role" phx-value-user_id={member.user_id}>` dropdown does not produce a server response. No "Role updated" flash message appears and the role badge remains unchanged. After navigating away and back, the DB value is also unchanged.

The issue was previously filed as qa-427 and marked resolved with the note "Wrapped role select in a form with hidden user_id input so phx-change sends the user_id in form data." However, the current implementation in `lib/metric_flow_web/live/account_live/members.ex` lines 92-110 does not use a form wrapper — it uses `phx-value-user_id` on the select element directly. Phoenix LiveView `phx-change` events do not include `phx-value-*` attributes in params — only `phx-click` events merge `phx-value-*` into params. As a result, the `change_role` handler receives `%{"role" => "..."}` without `user_id`, causing it to fail silently or hit the wrong clause.

Reproduction:
1. Log in as any admin or owner
2. Navigate to `/accounts/members`
3. Change any non-owner member's role in the dropdown
4. Observe: no flash message, no badge update, no DB change on reload

### QA database role drift: qa@example.com is admin instead of owner

#### Severity
MEDIUM

#### Scope
QA

#### Description
In the QA Test Account, `qa@example.com` holds the `admin` role instead of `owner`. `qa-member@example.com` is the current owner. This is the result of a previous test run that used `phx-click` on the hidden `[data-role="change-role"]` button (which does send `phx-value-user_id` correctly) and swapped the roles. The `qa_seeds.exs` script is idempotent but does not reset roles if the users already exist.

Impact on this test run:
- Scenario 5 could not verify owner-exclusive role options (`owner`, `admin` in invite form)
- Scenario 6 could not test inviting at `admin` role since the current user is not an owner
- Scenario 7 (`phx-change` role select) tested as admin, not owner - no functional difference but the brief assumes owner context

The seed script should reset the `qa@example.com` role to `owner` and `qa-member@example.com` to `read_only` (or member) on each run to restore the expected DB state. Alternatively, the seeds could drop and re-create the QA Test Account membership records.
