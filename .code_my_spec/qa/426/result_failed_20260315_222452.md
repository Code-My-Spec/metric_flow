## Issues

### Role change via inline select still drops LiveView connection (regression of qa-427)

#### Severity
HIGH

#### Scope
APP

#### Description
Selecting a new role from the inline `<select phx-change="change_role">` in the Actions column on `/accounts/members` causes the LiveView connection to drop with "We can't find the internet / Attempting to reconnect" errors. After page reload, the role is unchanged in the database. Reproduced twice during this test run.

Issue qa-427 was previously marked as resolved with the fix "Wrapped role select in a form with hidden user_id input". The fix is present in the code (the select IS inside a `<form phx-change="change_role">` with a hidden `<input name="user_id">`). However, both the `<form>` AND the `<select>` element have `phx-change="change_role"` attributes. When the select value changes, both events fire. This double-firing may be causing the LiveView connection to behave unexpectedly. The select-level `phx-change` fires first without form data (no user_id), hitting the handler which fails to match params, and the connection is disrupted before the form-level event delivers user_id.

Reproduction steps:
1. Log in as any admin or owner user
2. Navigate to `/accounts/members`
3. Find any non-owner member row
4. Change the role dropdown from the current value to a different value
5. Observe: LiveView shows reconnection errors, no "Role updated" flash, role unchanged on reload

### Admin invite form shows "admin" role option but backend rejects it

#### Severity
MEDIUM

#### Scope
APP

#### Description
When a user with the `admin` role visits `/accounts/members`, the invite form's role select (`select[name="invitation[role]"]`) shows `admin`, `account_manager`, `read_only`, and `member` as options. When the admin selects `admin` and submits the invite, the backend returns "You are not authorized to invite members".

The backend authorization is correct per `Authorization.can?/3`: admins cannot assign the `admin` role (only `account_manager` and below). But the UI invite form presents `admin` as a valid choice for admin users via `@admin_invite_roles ~w(admin account_manager read_only member)` in `members.ex`. The form should only show the roles the current user is authorized to assign (`account_manager`, `read_only`, `member` for admin users).

Reproduction steps:
1. Log in as a user who is `admin` (not `owner`) on an account
2. Navigate to `/accounts/members`
3. Fill the invite form with any existing user's email and select role `admin`
4. Click "Invite"
5. Observe: "You are not authorized to invite members" flash despite the form offering `admin` as an option

### QA seed data modified by previous test runs — qa@example.com is now admin, not owner

#### Severity
LOW

#### Scope
QA

#### Description
The seed script creates `qa@example.com` as the owner of "QA Test Account". However, prior QA runs have changed roles via the members page, resulting in `qa-member@example.com` being promoted to owner and `qa@example.com` being demoted to admin. The seed script is idempotent (it won't reset roles for existing accounts), so re-running it does not restore the expected owner state.

This affects Scenario 1 which expects `qa@example.com` to have the "owner" role badge, and Scenario 6 which requires an owner to successfully invite with the `admin` role (admin users cannot assign the admin role).

The seed script should either: (a) reset member roles when the users and account already exist, or (b) the brief should be updated to not assume qa@example.com is the owner.
