## Issues

### Bare "google" OAuth parent card visible in provider grid

#### Severity
MEDIUM

#### Scope
APP

#### Description
The `/integrations/connect` page shows a bare card with `data-platform="google"`, provider name "Google", empty description, and "Not connected" status. This card represents the OAuth parent provider and should not appear in the grid — only the child providers (google_ads, google_analytics, google_search_console) should be shown.

The fix (`MapSet.difference(@oauth_parent_keys)` in `build_provider_list/1`) is present in the current source code but the running server has not been restarted to pick up the change. Clicking the "Connect" button on this card initiates an OAuth flow to `accounts.google.com` with an unregistered redirect URI (`/integrations/oauth/callback/google`), causing a redirect_uri_mismatch error on Google's side.

Reproduced at: `http://localhost:4070/integrations/connect` — server restart required to verify the fix.

### Save account selection redirects to detail page instead of /integrations

#### Severity
LOW

#### Scope
APP

#### Description
After saving an account selection on `/integrations/connect/quickbooks/accounts`, the user is redirected to `/integrations/connect/quickbooks` (the detail view) instead of `/integrations` (the integrations list). The source code at `connect.ex` line 437 explicitly navigates to the provider detail page:

```elixir
push_navigate(to: ~p"/integrations/connect/#{provider_str}")
```

The spec (`connect.spec.md`) does not explicitly state where the redirect should go after account save, but the QA brief expected `/integrations`. The current behavior (returning to the detail page) may be intentional as it lets the user verify the saved selection before returning to the list. However, the brief's acceptance criteria diverge from implementation.

Reproduced at: `/integrations/connect/quickbooks/accounts` — fill in a company ID and click "Save Selection".

### No-params and failed-code OAuth callbacks redirect to grid instead of error detail view

#### Severity
LOW

#### Scope
APP

#### Description
When the OAuth callback receives no parameters (`/integrations/oauth/callback/quickbooks`) or a code that fails token exchange, the controller redirects to `/integrations/connect` (the provider grid) with a generic flash error "Could not complete the connection. Please try again." rather than redirecting to `/integrations/connect/quickbooks` with an error state that renders the "Connection Failed" detail view.

This inconsistency means:
- `error=access_denied` → renders Connection Failed detail view (correct per spec)
- `error=server_error` → renders Connection Failed detail view (correct per spec)
- No params or token exchange failure → redirects to grid with flash only (different UX path)

The user experience is less informative for the no-params and token-exchange-failure cases. Reproduced by navigating directly to `/integrations/oauth/callback/quickbooks` and `/integrations/oauth/callback/quickbooks?code=test_auth_code`.
