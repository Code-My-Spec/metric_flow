## Issues

### OAuth callback with invalid code or no code redirects to the connect index instead of the provider error view

#### Severity
HIGH

#### Description
When the OAuth callback fails due to missing or invalid code (internal error, not provider-reported `error=` parameter), the user is redirected to `/integrations/connect` (the provider selection index) instead of `/integrations/connect/quickbooks` (where the "Connection Failed" result view should appear).

Reproduced by navigating to:
- `http://localhost:4070/integrations/oauth/callback/quickbooks` (no params)
- `http://localhost:4070/integrations/oauth/callback/quickbooks?code=test_auth_code` (invalid code)

In both cases, the flash "Could not complete the connection. Please try again." appears as a toast on the provider selection index rather than inside the Connection Failed card on the provider detail page.

By contrast, OAuth callbacks with an explicit `error=` parameter (e.g., `error=access_denied` or `error=server_error`) correctly redirect to `/integrations/connect/quickbooks` and show the full Connection Failed UI.

Root cause appears to be in the error handling path: when the Assent OAuth2 strategy raises a `KeyError` (from accessing `session_params.state` on an empty map when no state was stored), the controller's rescue block runs and redirects to `/integrations/connect/quickbooks`. However, the browser consistently lands on `/integrations/connect`. This suggests either (a) the rescue block is not running and a higher-level Phoenix error handler is redirecting, or (b) the LiveView at `/integrations/connect/quickbooks` is itself redirecting to the index under some condition triggered by the flash state.

The "Connection Failed" UI at `/integrations/connect/quickbooks` is never shown for these failure paths, giving users no actionable error message or "Try again" link.

### Bare "google" provider card still appears in the connect grid

#### Severity
MEDIUM

#### Description
A `data-platform="google"` card with name "Google" and an empty description appears in the platform grid at `/integrations/connect`. This card represents the bare OAuth parent provider (`:google`) and should not be shown — only canonical provider cards (Google Ads, Google Analytics, Google Search Console) should appear.

The story notes indicate this card was supposed to be removed. Looking at `build_provider_list/1` in `connect.ex`, it builds the union of canonical providers + configured providers. The `:google` provider is in the configured providers returned by `Integrations.list_providers()` and is NOT in `@oauth_parent_keys = MapSet.new([:google])` exclusion... wait, it IS excluded via `MapSet.difference(@oauth_parent_keys)`. So configured `:google` is excluded. But the user has an existing integration with provider `:google` (from seed data), and `integration_keys` picks up `:google` from `Enum.map(integrations, & &1.provider)`. Since `integration_keys` is unioned without the parent key filter, the `:google` card reappears via the integration list path.

The filter `MapSet.difference(@oauth_parent_keys)` is only applied to `configured_keys`, not to `integration_keys`. Any existing `:google` integration record causes the bare card to reappear.

To reproduce: log in as qa@example.com and navigate to `http://localhost:4070/integrations/connect`. A "Google" card with empty description and `data-platform="google"` is visible.

### QA seeds include a pre-connected QuickBooks integration, preventing Scenario 12 verification

#### Severity
LOW

#### Scope
QA

#### Description
Scenario 12 attempts to verify that QuickBooks is NOT shown as connected in the integrations list before OAuth completes. However, the QA seed data (`priv/repo/qa_seeds.exs`) already creates a QuickBooks integration record for the QA account. As a result, QuickBooks is shown as "Connected" from the first test run.

The brief stated "The test creates no integration records prior to testing" — this is incorrect; the seed data does create a QuickBooks integration.

To properly test this scenario, the seed script should either not create a QuickBooks integration, or a teardown step should remove any existing QuickBooks integration before this scenario runs.
