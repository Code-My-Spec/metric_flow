## Issues

### Google bare OAuth parent key card appears in platform selection grid

#### Severity
MEDIUM

#### Description
The platform selection grid at `/integrations/connect` displays a "Google" card with `data-platform="google"`. This card represents the bare OAuth parent key `:google` which is supposed to be filtered out by `@oauth_parent_keys MapSet.new([:google])` in `build_provider_list/1`.

The filter removes `:google` from `configured_keys` (providers from Application env) and from `integration_keys` at the end, but it appears `configured_keys` or `integration_keys` may include `:google` in a way that survives into `all_keys`. The "Google" card is confusing to users because clicking it initiates a real OAuth flow to Google (reaching an OAuth error page about redirect_uri_mismatch) rather than showing an error flash.

Reproduced: Navigate to `http://localhost:4070/integrations/connect` while logged in as `qa@example.com`.

### Save Account Selection redirects to provider detail page, not /integrations

#### Severity
LOW

#### Description
The brief's Scenario 7 expects that saving an account selection redirects to `/integrations`. The actual implementation redirects to `/integrations/connect/{provider}` (the per-provider detail page). This is a discrepancy between the spec/brief and the implementation.

In `handle_event("save_account_selection", ...)` (connect.ex line 437), the push_navigate target is `~p"/integrations/connect/#{provider_str}"`. The spec says the OAuth result view should offer a "View Integrations" link at `/integrations`, but does not explicitly state the save-selection redirect target. Whether this is a bug or a brief error needs clarification.

Reproduced: Navigate to `/integrations/connect/quickbooks/accounts`, fill in a company ID, click "Save Selection".

### OAuth callback rescue clause redirects to /integrations/connect instead of per-provider detail

#### Severity
MEDIUM

#### Description
When the OAuth callback at `/integrations/oauth/callback/:provider` crashes (e.g., token exchange fails or params are missing), the `rescue` block in `IntegrationOauthController.callback/2` redirects to `~p"/integrations/connect"` (the selection grid). The expected behavior is a redirect to `~p"/integrations/connect/#{provider_str}"` (the per-provider detail page with a flash error).

This means the "Connection Failed" result view is NOT shown for these failure paths — the user instead sees the selection grid with a generic flash message ("Could not complete the connection. Please try again."). The per-provider result view only renders for error cases caught in `handle_oauth_callback/4` where an `"error"` param is present (access_denied, server_error).

Reproduced:
- Navigate to `http://localhost:4070/integrations/oauth/callback/quickbooks` (no params)
- Navigate to `http://localhost:4070/integrations/oauth/callback/quickbooks?code=test_auth_code` (invalid code that fails token exchange)

Fix: Change the rescue clause redirect target from `~p"/integrations/connect"` to `~p"/integrations/connect/#{provider_str}"`.

### QuickBooks "Connected as unknown" — no email in provider_metadata

#### Severity
INFO

#### Description
On the QuickBooks detail page (`/integrations/connect/quickbooks`), the connected user display shows "Connected as unknown". This is because `integration.provider_metadata["email"]` is nil for QuickBooks — QuickBooks OAuth does not return an email address as part of its token response, so the metadata key does not exist.

The UI defaults to "unknown" which is not user-friendly. A QuickBooks-specific label (e.g., "Company ID" or the realm ID) would be more informative. The Company ID IS shown below, so the information is technically available, but the "Connected as unknown" text is misleading.

Reproduced: Navigate to `http://localhost:4070/integrations/connect/quickbooks` after OAuth connection.
