## Issues

### No-params and failed-code OAuth callbacks redirect to grid instead of error detail view

#### Severity
LOW

#### Scope
APP

#### Description
When the OAuth callback receives no parameters (`/integrations/oauth/callback/quickbooks`) or a code that fails token exchange (`/integrations/oauth/callback/quickbooks?code=test_auth_code`), the controller redirects to `/integrations/connect` (the provider grid) with a generic flash error "Could not complete the connection. Please try again." rather than to `/integrations/connect/quickbooks` which would render the "Connection Failed" detail card.

This creates an inconsistency in error handling:
- `error=access_denied` → redirects to `/integrations/connect/quickbooks`, renders "Connection Failed" detail view (correct)
- `error=server_error` → redirects to `/integrations/connect/quickbooks`, renders "Connection Failed" detail view (correct)
- No params → redirects to `/integrations/connect` grid with flash only (inconsistent)
- Token exchange failure → redirects to `/integrations/connect` grid with flash only (inconsistent)

The user experience is less informative for the no-params and token-exchange-failure cases since the "Connection Failed" card with provider-specific "Try again" links is not shown.

Reproduced by navigating to:
- `http://localhost:4070/integrations/oauth/callback/quickbooks`
- `http://localhost:4070/integrations/oauth/callback/quickbooks?code=test_auth_code`

### Bare "google" OAuth parent card visible in provider grid

#### Severity
MEDIUM

#### Scope
APP

#### Description
The `/integrations/connect` page shows a bare card with `data-platform="google"`, provider name "Google", empty description, and a Connect button. This card represents the OAuth parent provider (`@oauth_parent_keys`) and should not appear in the grid — only the child providers (google_ads, google_analytics, google_search_console) should be shown.

Per task instructions, the fix is present in source code (`MapSet.difference(@oauth_parent_keys)` in `build_provider_list/1`) but the running server has not been restarted to pick it up. This issue is known and previously resolved — noted here as stale-server, not filed as a new bug requiring action.

Reproduced at: `http://localhost:4070/integrations/connect` — server restart required to verify the fix.
