## Issues

### Page heading and subtitle do not match spec

#### Severity
LOW

#### Description

The `IntegrationLive.Connect` page renders with heading "Connect a Provider" and subtitle
"Authenticate with your marketing providers to start syncing data". The spec
(`connect.spec.md`) describes the heading as "Connect a Provider" and subtitle as
"Authenticate with your marketing providers to start syncing data" — these match the spec.

However, the brief (Scenario 1) expected heading "Connect a Platform" and subtitle "Link
your marketing accounts to start syncing data". The brief is inconsistent with both the
spec and the running app. This is a QA brief authoring error.

#### Scope
QA

### Unexpected bare "google" provider card in the platform grid

#### Severity
MEDIUM

#### Description

A sixth card with `data-platform="google"` appears in the platform grid alongside the five
canonical providers. This card has no description, shows "Not connected", and clicking
"Connect" initiates a Google OAuth flow (distinct from `google_ads`, `google_analytics`,
etc.). This card appears because `Integrations.list_providers()` returns `:google` as a
configured key, and `build_provider_list/1` unions configured keys with canonical keys.
There is no entry for `:google` in `@canonical_providers` or `@provider_metadata`, so it
renders with a derived display name and empty description.

A bare "Google" card with no description alongside separate "Google Ads", "Google
Analytics", and "Google Search Console" cards is confusing — users do not know what
"Google" connects without a description. The `@provider_metadata` map should include an
entry for `:google`, or the `list_providers()` return should be filtered to exclude the
base `google` OAuth key from the selection grid UI.

### OAuth callback with no parameters or invalid code redirects to grid instead of provider error view

#### Severity
MEDIUM

#### Description

When the OAuth callback receives no parameters (`/integrations/oauth/callback/quickbooks`)
or a code with no matching state (`?code=test_auth_code`), the controller redirects to
`/integrations/connect` (the platform selection grid) with a generic flash error "Could
not complete the connection. Please try again."

The `render_result/1` view with "Connection Failed" heading is only shown when the
controller redirects to `/integrations/connect/:provider` with an error flash. The
no-params and invalid-state paths do not include the provider in the redirect, so the
error view is never rendered for these cases.

Users lose context of which provider failed and see a generic error on an unrelated page.
The controller should redirect to `/integrations/connect/quickbooks` (with the provider
param) even for no-params and state-mismatch errors, so the "Connection Failed" view
appears with "Try again" and "Back to integrations" links.

Reproduced at:
- `http://localhost:4070/integrations/oauth/callback/quickbooks` (no params)
- `http://localhost:4070/integrations/oauth/callback/quickbooks?code=test_auth_code` (code, no matching state)

### Save account selection redirects to provider detail, not /integrations

#### Severity
LOW

#### Description

After successfully saving an account selection, `handle_event("save_account_selection")`
calls `push_navigate(to: ~p"/integrations/connect/#{provider_str}")`, which navigates to
the provider detail page (e.g., `/integrations/connect/quickbooks`). The spec
(Scenario 7 brief) expects a redirect to `/integrations`. The current behavior is not
wrong per the spec document (`connect.spec.md`), but it differs from the acceptance
criterion "After saving, user is redirected to /integrations" stated in the brief.

If the intent is for the save to complete the connection flow and land the user on the
integrations list, the redirect target should be `/integrations`.

### Scenario 2 (unsupported_platform) not testable — card absent from grid

#### Severity
LOW

#### Scope
QA

#### Description

Scenario 2 in the brief tests clicking a `[data-platform='unsupported_platform']` card
that is described as being "shown in the grid". No such card exists in the running app.
The platform grid only shows canonical and configured providers. The brief's setup notes
say "The `unsupported_platform` card IS shown in the grid" — this is incorrect. The
feature was likely removed or never implemented as described.

The brief should either remove this scenario or note that the unsupported_platform card
is not present and the scenario cannot be tested.

### QuickBooks integration pre-exists in QA seeds, preventing Scenario 12 verification

#### Severity
LOW

#### Scope
QA

#### Description

The QA seeds create a QuickBooks integration for the `qa@example.com` account, causing
Scenario 12 ("Verify QuickBooks is NOT shown as Connected before OAuth") to fail
immediately. The precondition "no integration records prior to testing" in the brief
conflicts with the actual seed state.

To test the acceptance criterion "Integration is saved only after successful OAuth
completion", either:
1. The seeds should not create a QuickBooks integration, or
2. A separate test account with no integrations should be used for this scenario.
