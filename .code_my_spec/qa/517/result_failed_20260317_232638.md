## Issues

### Sync schedule description does not mention Google Business Profile

#### Severity
MEDIUM

#### Description
The automated sync schedule description in `[data-role="sync-schedule"]` lists covered providers as "Google Ads, Facebook Ads, Google Analytics" (marketing) and "QuickBooks" (financial), but does not mention Google Business Profile.

Story 517 adds Google Business Profile as a sync provider, but the static schedule description paragraph in `SyncHistory` was not updated. Users have no indication from the schedule section that Google Business Profile data is synced daily.

The BDD spec criterion_4819 (`the schedule section lists Google Business Profile as a covered provider`) fails because of this omission.

Source location: `lib/metric_flow_web/live/integration_live/sync_history.ex`, line 48–53.

Fix: Update the schedule description to include "Google Business Profile" in the marketing providers list, e.g.: "Covers marketing providers (Google Ads, Facebook Ads, Google Analytics, Google Business Profile) and financial providers (QuickBooks)."

### google_business and google_business_reviews absent from @provider_names map

#### Severity
LOW

#### Description
`:google_business` and `:google_business_reviews` are not in the `@provider_names` module attribute in `SyncHistory`. Both providers fall through to `derive_display_name/1`, which produces acceptable display names ("Google Business" and "Google Business Reviews").

This is not a user-visible bug — the display names are correct via the fallback. However, the pattern is inconsistent: all other providers are explicitly named. Adding them to `@provider_names` would be cleaner and more maintainable.

Source location: `lib/metric_flow_web/live/integration_live/sync_history.ex`, lines 20–27.

### Seed script does not clear sync history — empty state cannot be tested via browser

#### Severity
LOW

#### Scope
QA

#### Description
The standard QA seed script (`priv/repo/qa_seeds.exs`) does not truncate or clear the `sync_history` table. After multiple QA runs, the seeded account accumulates sync history entries, making it impossible to test the empty state (`[data-role="sync-history"]` showing "No sync history yet.") via browser automation.

The empty state is verified by the BDD spec suite instead, but the brief states it should be testable via browser. A story-specific seed or a seed script that clears sync history for the QA account would allow browser-based empty state testing.

Workaround: The BDD specs in `test/spex/517_*/` create isolated test processes that start with no history and test the empty state path.
