## Issues

### Invite form defaults to "owner" role when no explicit selection is made

#### Severity
HIGH

#### Description
When the invite form is submitted without explicitly changing the role select, the invited user receives the `owner` role. The first `<option>` in `select[name='invitation[role]']` is `owner` with no `selected` attribute directing the form to a safer default.

Reproduced: filled only `input[name='invitation[email]']` with `qa-member@example.com` and clicked "Invite" without touching the role select. Result: `qa-member@example.com` was added as `owner`, as shown by the "owner" badge in the members list.

This is a significant UX safety risk: an account owner who does not notice the role select will accidentally grant full owner-level access to every invited user. The invite form's default should be `read_only` or `member`, not `owner`. The `@owner_invite_roles` list in the source is `~w(owner admin account_manager read_only member)` — reordering it to place `read_only` first (or adding a `selected` default in the template) would fix the issue.

Reproduction steps:
1. Log in as `qa@example.com`, navigate to `/accounts/members`
2. Fill `input[name='invitation[email]']` with a valid user email
3. Do not change the role select — leave it on its default (first) value
4. Click "Invite"
5. Observe the invited user has `owner` role in the members list
