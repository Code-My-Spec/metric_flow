## Issues

### Brief uses incorrect selector for role change select

#### Severity
LOW

#### Scope
QA

#### Description
Scenario 4 in the brief specifies `select[phx-change='change_role']` as the selector for the role change dropdown. The actual implementation in `members.ex` uses a `<form phx-submit="change_role">` wrapping each row's select. The select element has `name="role"` and no `phx-change` attribute. Interaction requires selecting the value in `select[name='role']` and then clicking the "Change" `button[type='submit']` within the same form. The brief's selector will not match and will cause automation failures if used literally.

The correct approach is to target the select by row: `[data-role='member-row']:nth-child(N) select[name='role']` and then click the corresponding submit button.

### Seeds cannot be re-run while Phoenix server is running (role drift between QA sessions)

#### Severity
MEDIUM

#### Scope
QA

#### Description
`mix run priv/repo/qa_seeds.exs` fails when the Phoenix server is running because the Cloudflare tunnel GenServer conflicts with the application startup. This is documented in the QA plan, but the consequence for Story 430 is significant: role changes made during a QA session persist to the next session. In this run, `qa-member@example.com` started as `owner` in "QA Test Account" (from a prior role-change test) rather than the expected `read_only`. This caused Scenario 3 (invite flow) to be only partially testable and forced Scenario 6 to use a workaround.

The QA plan notes an alternative invocation using `--no-start`, but the alternative also requires the application modules to be started manually in the correct order. A `MIX_ENV`-gated reset endpoint or a Wallaby/ExUnit sandbox reset would resolve this for live-server QA sessions.

### Member's active account is not the account they were invited to (primary_account picks most recent)

#### Severity
LOW

#### Scope
APP

#### Description
When `qa-member@example.com` logs in and navigates to `/accounts/members`, the LiveView mounts the account returned by `ActiveAccountHook.primary_account/1`, which selects the most recently active account. Because the member has pre-existing memberships in multiple accounts (from prior QA runs), their session loaded "Client Alpha" rather than "QA Test Account". As a result, Scenario 6 (immediate access revocation) could not verify the expected behavior: removing a member from QA Test Account does not affect their ability to view "Client Alpha" data.

This is expected behavior for the current implementation, but it means that access revocation is only immediate if the member's active session is on the account they are being removed from. If the member is viewing a different account, they retain uninterrupted access to that account. This is by design (each account has its own membership), but testers and users should be aware that "immediate revocation" applies only to the specific account, not all accounts the user belongs to.
