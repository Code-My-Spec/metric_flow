## Issues

### Seed script does not clear sync history before each QA run

#### Severity
MEDIUM

#### Scope
QA

#### Description
The brief states: "Google integration with provider `:google` and no sync history (cleared)." However, the base `priv/repo/qa_seeds.exs` does not clear `sync_jobs` records. After multiple QA sessions the sync history table accumulates entries, making it impossible to test the empty state without a manual database clear or a story-specific seed script.

Scenario 3 (empty state verification) could not be executed because the QA Test Account had many existing sync history entries. A QA seed script specific to story 516 should truncate `sync_jobs` for the QA account before the test runs, or the base seeds should include a history-clearing step.

### Brief incorrectly describes @provider_names map as missing :google_search_console

#### Severity
LOW

#### Scope
QA

#### Description
The brief (`.code_my_spec/qa/516/brief.md`) repeatedly states that `:google_search_console` is NOT in the `@provider_names` map and "falls through to `derive_display_name/1`". Source code inspection of `sync_history.ex` lines 20–27 shows `:google_search_console` IS explicitly present in the map with value "Google Search Console". Scenario 5 and Scenario 6 of the brief are premised on this being a gap, but the gap does not exist. The BDD spex tests confirm the provider is handled correctly.

The brief should be corrected to reflect that the provider is explicitly registered, and Scenarios 5 and 6 should be updated accordingly.

### Sync job error messages truncated to 255 characters causing PostgreSQL error 22001

#### Severity
HIGH

#### Scope
APP

#### Description
The sync history page shows multiple failed entries with error message `ERROR 22001 (string_data_right_truncation) value too long for type character varying(255)`. This means the `SyncWorker` or `DataSync` context is attempting to persist an error message that exceeds 255 characters into a `varchar(255)` column in the `sync_jobs` table. Instead of gracefully truncating the message or using a `text` column, the insert fails at the database layer with this Postgres error.

The visible error message text in the UI for these entries is the database exception itself, not the original sync failure reason. The actual sync error is lost.

Reproduced by observing existing sync history entries on `http://localhost:4070/integrations/sync-history` for the QA Test Account. Multiple affected entries with `data-status="failed"` show this error text in `[data-role="sync-error"]`. The error_message column on the sync_jobs table should either be changed to `text` (no length limit) or the application should truncate error messages to 255 characters before persisting.
