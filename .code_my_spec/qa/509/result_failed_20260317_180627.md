## Issues

### criterion_4761 spex fails to compile due to overly long test description

#### Severity
MEDIUM

#### Scope
QA

#### Description
The file `test/spex/509_sync_google_analytics_4_data/criterion_4761_ga4_runreport_10_metric_limit_chunked_fetch_and_merge_by_date_spex.exs` fails to compile with an Erlang atom table overflow error when the Elixir compiler tries to derive a function name from the spex description string. The description is:

> "Because GA4's runReport endpoint has a 10-metric limit per request, metrics are fetched in chunks of 10 and results are merged by date before storage; the final stored data is equivalent to what a single-request response would contain"

Erlang internally calls `list_to_atom/1` on this string to generate an internal function name for the test, which exceeds the system atom table limit. The stack trace shows the failure in `v3_core.erl` during compilation.

Steps to reproduce: `mix test test/spex/509_sync_google_analytics_4_data/criterion_4761_ga4_runreport_10_metric_limit_chunked_fetch_and_merge_by_date_spex.exs`

Fix: Shorten the `spex "..."` description string in the module. For example: `spex "GA4 runReport 10-metric limit: results from chunked fetches are merged by date before storage"`. The test logic itself (scenarios) is correct and covers the required behavior.

### Persisted sync history entries show "Google" instead of platform-specific names

#### Severity
MEDIUM

#### Scope
APP

#### Description
In the Sync History page, persisted `SyncJob` records from the database show "Google" as the provider name for entries that are stored with `provider: :google`. This occurs because the `sync_worker.ex` uses `:google` as the SyncJob provider key when running the combined Google provider (which covers both Google Ads and Google Analytics). The `SyncHistory` LiveView's `@provider_names` map maps `:google` to `"Google"`, so those entries display ambiguously as "Google" instead of identifying the specific platform.

Observed in the sync history list at `http://localhost:4070/integrations/sync-history`: multiple entries labeled "Google" for both success and failed states. The spec says entries should show "provider name" — but the persisted success entries show only "Google" rather than "Google Analytics" or "Google Ads", making the sync history ambiguous.

This is particularly confusing alongside entries labeled "Facebook Ads" (which use a platform-specific name). Users cannot tell from the sync history whether a "Google" sync entry corresponds to Google Ads, Google Analytics, or both.

### sync_jobs table retains stale records from previous QA runs (empty state not testable)

#### Severity
LOW

#### Scope
QA

#### Description
The QA database contains sync job records from previous test runs, making Scenario 4 (empty state verification) untestable without manual database cleanup. The `qa_seeds.exs` script does not truncate the `sync_jobs` table, so the empty state (`No sync history yet.`) is never visible in repeated QA runs.

To fix, add a cleanup step to `priv/repo/qa_seeds.exs` that deletes existing sync_jobs for the QA account before inserting seed data, or provide a separate cleanup script. Alternatively, document that Scenario 4 requires a fresh database or manual truncation of `sync_jobs`.
