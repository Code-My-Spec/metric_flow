## Issues

### Save name input phx-change event does not update server-side state

#### Severity
HIGH

#### Description
The `[data-role="save-name-input"]` input has `phx-change="update_save_name"` but the event never fires to update the server-side `save_name` assign. When a user types in the save name field, the browser DOM shows the typed text visually, but the LiveView server-side state keeps `save_name: ""`. As a result, the "Save Visualization" button remains permanently disabled after chart generation — users cannot save any generated visualization.

Reproduction steps:
1. Log in as `qa@example.com`
2. Navigate to `http://localhost:4070/reports/generate`
3. Type a prompt and click "Generate Chart"
4. Wait for the chart preview to appear (save section becomes visible)
5. Type any text into the "Visualization name" input
6. Observe: the "Save Visualization" button remains disabled; the server-side HTML `value=""` attribute on the input does not change

Root cause: The input is a standalone `<input>` not inside a `<form>`. Without a `phx-debounce` attribute, LiveView's `phx-change` on a standalone input relies on the DOM `change` event (blur). Neither blur (Tab key) nor Playwright's fill/type/keys triggered the event. Adding `phx-debounce="blur"` or wrapping the input in a form may resolve this.

Source file: `lib/metric_flow_web/live/ai_live/report_generator.ex`, line 103.

### "View in Visualizations" link navigates to /dashboards, not /visualizations

#### Severity
MEDIUM

#### Description
The save confirmation section's "View in Visualizations" link (in `lib/metric_flow_web/live/ai_live/report_generator.ex`, line 135) navigates to `/dashboards`, not `/visualizations`. The brief specifies the link should point to `/visualizations`. The `/visualizations` route does not exist in the router — a direct GET returns `Phoenix.Router.NoRouteError`. If the intent is for the link to point to a visualizations index page, that route needs to be added, or the link destination needs to be updated to the appropriate existing route (`/dashboards`).

Reproduction: Review source at line 135 — `<.link navigate="/dashboards" class="link block mb-3">View in Visualizations</.link>`. Alternatively, navigate to `http://localhost:4070/visualizations` to see the NoRouteError.
