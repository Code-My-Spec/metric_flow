# Authorization Strategy: Patterns and Usage

MetricFlow uses a hand-rolled, scope-based authorization system generated by `phx.gen.auth`.
There is no external authorization library. All checks flow through `MetricFlow.Accounts.Authorization`
and are enforced inside context functions before any data access or mutation occurs.

See the decision record at `docs/architecture/decisions/authorization_strategy.md` for why
this approach was chosen over libraries like LetMe, Bodyguard, or Permit.

---

## 1. The Scope Struct

`MetricFlow.Users.Scope` is the central carrier of the calling user's identity and active context.
It is built once per request or LiveView mount and threaded as the first argument through all
context function calls.

### Definition

```elixir
# lib/metric_flow/users/scope.ex
defmodule MetricFlow.Users.Scope do
  defstruct user: nil,
            active_account: nil,
            active_account_id: nil,
            active_project: nil,
            active_project_id: nil

  @type t :: %__MODULE__{
          user: User.t() | nil,
          active_account: any(),
          active_account_id: String.t() | nil,
          active_project: any(),
          active_project_id: String.t() | nil
        }
end
```

### Key Fields

| Field | Type | Purpose |
|---|---|---|
| `user` | `%User{}` | The authenticated user making the request |
| `active_account` | `%Account{} \| nil` | The preloaded account struct (set from user preferences) |
| `active_account_id` | `String.t() \| nil` | The UUID of the active account |
| `active_project` | `%Project{} \| nil` | The preloaded active project (future use) |
| `active_project_id` | `String.t() \| nil` | The UUID of the active project |

### Building a Scope

```elixir
# Basic scope from a user (no account context)
scope = Scope.for_user(user)

# Full scope with account and project preferences loaded
# This is done by the web layer, not the domain layer
scope = Scope.for_user(user) |> UserPreferences.load_into_scope()
```

`Scope.for_user/1` returns `nil` when passed `nil`, which allows safe use in the
unauthenticated `live_session :current_user` route group.

### Accessing the Current User in Templates

In LiveViews and templates, always access the current user through the scope:

```heex
<%!-- CORRECT --%>
{@current_scope.user.email}

<%!-- FORBIDDEN - @current_user is never assigned --%>
{@current_user.email}
```

### Scope in the Router

`phx.gen.auth` assigns `@current_scope` in both plugs and LiveView sockets:

- `fetch_current_scope_for_user` — Populates scope if a session cookie is present (optional auth)
- `require_authenticated_user` — Same, but redirects to `/users/log-in` if unauthenticated

All authenticated LiveViews sit inside the `live_session :require_authenticated_user` block.
The `on_mount: [{MetricFlowWeb.UserAuth, :require_authenticated}]` callback performs the
redirect check and populates `socket.assigns.current_scope`.

## 2. The Authorization Module

`MetricFlow.Accounts.Authorization` contains all authorization logic for account-level operations.

### Source

```elixir
# lib/metric_flow/accounts/authorization.ex
defmodule MetricFlow.Accounts.Authorization do
  alias MetricFlow.Accounts.MembersRepository
  alias MetricFlow.Users.Scope

  def authorize(action, %Scope{} = scope, resource) do
    case {action, resource} do
      {:read_account, account_id} when is_binary(account_id) ->
        MembersRepository.user_has_account_access?(scope.user.id, account_id)

      {:manage_account, account_id} when is_binary(account_id) ->
        user_role = MembersRepository.get_user_role(scope.user.id, account_id)
        user_role in [:owner, :admin]

      {:manage_members, account_id} when is_binary(account_id) ->
        user_role = MembersRepository.get_user_role(scope.user.id, account_id)
        user_role in [:owner, :admin]

      {:delete_account, account_id} when is_binary(account_id) ->
        user_role = MembersRepository.get_user_role(scope.user.id, account_id)
        user_role == :owner

      _ ->
        false
    end
  end

  def authorize!(action, scope, resource) do
    unless authorize(action, scope, resource) do
      raise "Unauthorized: #{action} on #{inspect(resource)}"
    end
  end
end
```

### `authorize/3` vs `authorize!/3`

| Function | Returns | Use when |
|---|---|---|
| `authorize/3` | `true` / `false` | You need to branch on the result (e.g., `get_account/2` returning `nil` instead of raising) |
| `authorize!/3` | `:ok` or raises | The caller must have permission — an unauthorized call is a programming error or security violation |

### Defined Actions

| Action | Resource | Who can perform it |
|---|---|---|
| `:read_account` | `account_id` | Any member of the account (owner, admin, member) |
| `:manage_account` | `account_id` | Owners and admins |
| `:manage_members` | `account_id` | Owners and admins |
| `:delete_account` | `account_id` | Owner only |

Any unrecognized `{action, resource}` pair returns `false` via the catch-all clause.

### Adding a New Action

Extend the `case` clause in `authorize/3`. For example, to add a `:transfer_ownership` action:

```elixir
{:transfer_ownership, account_id} when is_binary(account_id) ->
  user_role = MembersRepository.get_user_role(scope.user.id, account_id)
  user_role == :owner
```

Actions are plain atoms — there is no registry or macro to update. The pattern match is the
complete list of what is supported.

## 3. Role Hierarchy

The `members` table stores a role for each `(user_id, account_id)` pair. Roles are an Ecto enum:

```elixir
field :role, Ecto.Enum, values: [:owner, :admin, :member], default: :member
```

### Permissions by Role

| Capability | owner | admin | member |
|---|---|---|---|
| Read account data (dashboards, integrations, metrics) | yes | yes | yes |
| Edit account settings | yes | yes | no |
| Add/remove/update members | yes | yes | no |
| Delete the account | yes | no | no |

### Hierarchy Check (Member Schema)

`Member.has_role?/2` implements a numeric hierarchy check for situations where you need
"at least X role":

```elixir
# lib/metric_flow/accounts/member.ex
def has_role?(member, required_role) do
  role_hierarchy = %{member: 1, admin: 2, owner: 3}
  Map.get(role_hierarchy, member.role, 0) >= Map.get(role_hierarchy, required_role, 0)
end

# Convenience predicates
def owner?(member), do: member.role == :owner
def admin_or_owner?(member), do: member.role in [:admin, :owner]
```

These predicates are on the `Member` schema. They are used directly when you already have
a `%Member{}` struct loaded. The `Authorization` module uses `MembersRepository.get_user_role/2`
when you only have `user_id` and `account_id` available (which is the common case in contexts).

### Business Rules Enforced by the Data Layer

- An account must always have at least one owner.
  - `MembersRepository.remove_user_from_account/2` returns `{:error, :last_owner}` when the
    last owner is removed.
  - `Member.validate_owner_exists/2` prevents role downgrades that would leave no owners.
- A user can only have one membership per account (enforced by a unique index on `[user_id, account_id]`).
- Users are added as `:member` by default when no role is specified.

## 4. Context Function Patterns

### The Contract

Every public context function that touches account-scoped data follows this contract:

```
def function_name(%Scope{} = scope, ...) do
  # 1. Authorize (raise or branch)
  # 2. Perform data operation
  # 3. Broadcast PubSub message (if mutation)
  # 4. Return {:ok, result} | {:error, reason}
end
```

### Pattern A: Gate-and-proceed with `authorize!/3`

Use when an unauthorized call is a clear violation. The function raises immediately and
the LiveView error boundary handles it.

```elixir
def update_account(%Scope{} = scope, %Account{} = account, attrs) do
  Authorization.authorize!(:manage_account, scope, account.id)

  with {:ok, account} <- AccountsRepository.update_account(account, attrs) do
    broadcast_account(scope, {:updated, account})
    {:ok, account}
  end
end
```

### Pattern B: Silent nil return with `authorize/3`

Use for read functions where "no access" and "not found" should look the same to the caller
(prevents leaking the existence of resources to unauthorized users).

```elixir
def get_account(%Scope{} = scope, id) do
  case AccountsRepository.get_account(id) do
    nil ->
      nil

    account ->
      case Authorization.authorize(:read_account, scope, account.id) do
        false -> nil
        true -> account
      end
  end
end
```

### Pattern C: Bang variant with `authorize!/3`

For `get_account!/2`, where the caller expects to receive the account or have an exception raised:

```elixir
def get_account!(%Scope{} = scope, id) do
  account = AccountsRepository.get_account!(id)
  Authorization.authorize!(:read_account, scope, account.id)
  account
end
```

### Scope as First Parameter

The `%Scope{}` pattern match in the function head is not just convention — it is enforced
by the type system and will raise a `FunctionClauseError` if the wrong type is passed.
This is a deliberate design choice: passing something other than a `Scope` to a context
function is always a bug.

```elixir
# Correct
Accounts.get_account(scope, account_id)

# Will raise FunctionClauseError — intentional
Accounts.get_account(%{user: user}, account_id)
```

### Authorization in Sub-Contexts

Not all contexts apply their own authorization checks. The `Integrations` context, for example,
scopes all queries to `scope.user.id` at the repository level but does not call `Authorization`
because integrations are user-level resources, not account-level ones. When a context is
account-scoped, always apply the appropriate `authorize/3` or `authorize!/3` call before
any data operation.

## 5. Testing Authorization in Context-Level Unit Tests

Authorization is tested through the context interface, not by calling `Authorization` directly.
The pattern is:

1. Set up fixtures for the appropriate role (owner, admin, or member).
2. Build a scope from the fixture user using `user_scope_fixture/1`.
3. Call the context function and assert on the outcome.

### Setup

```elixir
use MetricFlowTest.DataCase
import MetricFlowTest.AccountsFixtures
import MetricFlowTest.UsersFixtures
```

### Testing That an Authorized Role Succeeds

```elixir
test "owner can update account settings" do
  owner = user_fixture()
  scope = user_scope_fixture(owner)
  account = account_with_owner_fixture(owner)

  assert {:ok, updated} = Accounts.update_account(scope, account, %{name: "New Name"})
  assert updated.name == "New Name"
end

test "admin can update account settings" do
  owner = user_fixture()
  owner_scope = user_scope_fixture(owner)
  account = account_with_owner_fixture(owner)

  admin = user_fixture()
  admin_scope = user_scope_fixture(admin)
  {:ok, _} = Accounts.add_user_to_account(owner_scope, admin.id, account.id, :admin)

  assert {:ok, _} = Accounts.update_account(admin_scope, account, %{name: "New Name"})
end
```

### Testing That an Unauthorized Role is Rejected

Use `assert_raise RuntimeError` to capture the `authorize!` exception. The message will contain
the action name and resource.

```elixir
test "member cannot update account settings" do
  owner = user_fixture()
  owner_scope = user_scope_fixture(owner)
  account = account_with_owner_fixture(owner)

  member = user_fixture()
  member_scope = user_scope_fixture(member)
  {:ok, _} = Accounts.add_user_to_account(owner_scope, member.id, account.id, :member)

  assert_raise RuntimeError, ~r/Unauthorized/, fn ->
    Accounts.update_account(member_scope, account, %{name: "Unauthorized Change"})
  end
end
```

### Testing Silent nil Returns (Pattern B)

When `authorize/3` is used (not `authorize!/3`), an unauthorized call returns `nil`.
Test this by verifying the return value, not by catching an exception.

```elixir
test "user without account access cannot read account" do
  owner = user_fixture()
  account = account_with_owner_fixture(owner)

  outsider = user_fixture()
  outsider_scope = user_scope_fixture(outsider)

  assert nil == Accounts.get_account(outsider_scope, account.id)
end
```

### Using Fixtures for Role Setup

The test fixtures provide helpers for common setup patterns:

```elixir
# Create an account and a member with the given role in one step
account = account_with_owner_fixture(user)          # user is owner
member_fixture(user2, account, :admin)              # user2 is admin
member_fixture(user3, account, :member)             # user3 is member (default)

# Build a scope for any user
scope = user_scope_fixture(user)

# Build a scope with an active account set
scope = user_scope_fixture(user, account)
```

`user_scope_fixture(user, account)` sets `active_account_id` and `active_account` on the
scope. Use this when the function under test reads `scope.active_account_id`.

### Test File Conventions

Authorization tests belong in the context test file alongside other function tests. Group
them in the same `describe` block as the function being tested:

```elixir
describe "update_account/3" do
  test "updates account when user is owner" do ... end
  test "updates account when user is admin" do ... end
  test "raises when user is member" do ... end
  test "raises when user has no account access" do ... end
  test "returns error with invalid attributes" do ... end
end
```

This keeps authorization verification co-located with the happy-path tests so they are not
forgotten when the function signature changes.

## 6. Quick Reference

### Authorization Check by Context Function

| Function | Check | Fails with |
|---|---|---|
| `get_account/2` | `:read_account` | Returns `nil` |
| `get_account!/2` | `:read_account` | Raises `RuntimeError` |
| `update_account/3` | `:manage_account` | Raises `RuntimeError` |
| `delete_account/2` | `:manage_account` | Raises `RuntimeError` |
| `change_account/3` | `:manage_account` | Raises `RuntimeError` |
| `list_account_members/2` | `:read_account` | Raises `RuntimeError` |
| `add_user_to_account/4` | `:manage_account` | Raises `RuntimeError` |
| `remove_user_from_account/3` | `:manage_members` | Raises `RuntimeError` |
| `update_user_role/4` | `:manage_members` | Raises `RuntimeError` |
| `get_user_role/3` | `:read_account` | Raises `RuntimeError` |

### When to Add a New Action vs. Reuse an Existing One

- Add a new action when the permission is meaningfully different from existing ones and you
  want to be able to grant it independently in the future.
- Reuse `:manage_account` for general owner+admin operations that fit the "managing settings"
  concept.
- Reuse `:manage_members` for anything that modifies who has access to the account.
- Always use `:delete_account` for destructive operations that should be owner-only, even if
  the underlying resource is not the account itself (e.g., bulk-deleting all account data).
