# MetricFlow: The Good, The Bad, and The Ugly

**A case study in building a full-stack application with a software factory — zero human-written code, 40 commits, 13 working days.**

---

MetricFlow is a Phoenix LiveView marketing analytics platform that correlates advertising spend across Google Ads, Facebook Ads, and Google Analytics with revenue data from QuickBooks. It offers multi-tenant accounts with role-based access, OAuth integrations for five platforms, correlation analysis with AI-powered insights, interactive Vega-Lite dashboards, and an AI chat interface for data exploration. The entire codebase — every module, every test, every migration — was generated by a software factory called CodeMySpec, orchestrated by Claude Code agents working from specifications. No human wrote a single line of production code.

The calendar span is January 30 to March 20, 2026 — but the actual work happened across just 13 days: one day for the scaffold, a two-day sprint for specs and core domain, and then ten days of implementation, QA, and fixes spread across March. Here's an honest assessment.

---

## The Good

The velocity was real. The first commit landed on January 30, 2026 — a Phoenix scaffold with `phx.gen.auth` and docs. Two days later, on February 22-23, the factory laid down the entire foundation in a 48-hour sprint: Oban job processing, Vault secret management, PromEx observability, the full BDD framework, and 25,000+ lines of specifications before a single line of business logic existed. By the end of February, commit `7ab7a8f` delivered the core domain — Accounts, Integrations, Metrics, DataSync with four data providers, and ~22,000 lines of unit tests plus 50+ BDD spex files.

The broad strokes were correct. The architecture held up: Boundary-enforced module dependencies, context modules as the public API surface, `%Scope{}` threaded through every public function, repository pattern separating queries from business logic. The factory understood Phoenix conventions and produced idiomatic Elixir. When it generated the invitation system (`0b7620b`, `d3a6da4`), it built the full lifecycle — send, accept, cancel, role assignment, token invalidation — with appropriate authorization checks. When it built the correlation analysis engine, it produced both raw and SmartAI modes with goal configuration, platform filtering, and feedback collection.

The dashboard redesign (`6de30d6`) is a good example of what the factory can do well when the spec is clear. It generated a multi-series Vega-Lite chart with date range controls, platform filters, data tables, and proper Phoenix hooks for CSP-safe rendering. The `QueryBuilder` (`7fd29c6`) that feeds it was clean, tested, and worked on the first deploy.

The QA infrastructure was arguably the factory's best output. Over the final two weeks, automated Vibium browser agents ran QA scenarios against 25+ stories, producing hundreds of screenshots and structured pass/fail reports. Stories 424 through 518 each have a `qa/` directory with briefs, result files, and screenshot evidence. The QA agents found real bugs — nil account guards, broken token invalidation, missing navigation links, role select defaults — and the coding agents fixed them in the same commit cycle (`d3a6da4` alone addresses a dozen QA-discovered issues).

The factory generated 100% of the code. I didn't write any of it. It did 90% of the orchestration and prompting without my intervention. That's the headline number.

## The Bad

When the application came off the line, a meaningful number of features were anemic or broken in ways that required manual intervention to diagnose.

**The agency-to-account link was broken.** The white-label and agency features were structurally present but not wired together. Commits `0973267` and `88170e8` — the last two commits in the repo, written with heavy human guidance — finally connected agency accounts to client accounts and got the auto-enrollment working. The factory had generated the `Agencies` context, the `AgenciesRepository`, and the `WhiteLabelHook`, but never connected them to the account settings UI.

**QuickBooks data sync was wrong.** The original QuickBooks provider fetched data in a format that didn't match how the correlation engine consumed it. Commit `f4661c2` is an 8,000-line rewrite of the QuickBooks provider — switching from aggregate fetches to daily credit/debit breakdowns. The factory's first attempt produced code that technically synced data but produced unusable correlation results. The cassette fixtures alone went from 352 lines to 7,708 lines to capture real QuickBooks sandbox responses.

**Visualizations were anemic.** The `ReportGenerator` (`7fd29c6`) generates Vega-Lite specs from natural language prompts via an LLM, but the generated reports are generic and not anchored to the user's actual data. You can't refine them by conversation. The saved visualizations have no useful index page — story 452's QA run needed four failed attempts before the route even existed (`0d35662` added it). The feature technically works but wouldn't survive contact with a real user.

**AI chat doesn't have access to real data.** The chat interface (`451`) renders, accepts messages, saves history, and can technically call the LLM — but the context injection for metrics, correlations, and dashboards is stubbed. The QA agent marked it as passing because it could send messages and get responses. The responses just aren't grounded in anything useful.

**SmartAI correlations are incomplete.** The top-5 positive / top-5 negative correlation display that the spec calls for doesn't actually filter and rank correctly. Story 449's QA screenshots show the panel rendering but the data behind it isn't right.

These are all things a human would catch in five minutes of clicking around. The QA agents either didn't catch them or marked them as passing because the UI elements existed.

## The Ugly

The integrations were a nightmare, and they exposed the deepest flaw in the factory model.

The integration code — OAuth flows for Google Ads, Google Analytics, Facebook Ads, QuickBooks, and Google Search Console — went through more QA iterations than every other feature combined. Story 434 (connect marketing platform via OAuth) has **six** failed QA runs before passing. Story 436 (disconnect/reconnect integrations) has **seven** failed runs. Story 438 (trigger data sync) has **twelve** failed runs across two days.

Here's what happened: the factory would generate integration code, the QA agent would test it, find it broken, the coding agent would "fix" it, and the QA agent would report it as passing. Then I'd go look at the integrations myself and everything was still broken. The QA agent told me over and over that QA passed, and I would go in and click through the integrations and find that nothing actually worked.

The problem was compounding failures. When the agent has to fight with both the integration code being wrong AND the OAuth secrets/tokens being invalid or misconfigured at the same time, it just gives up and papers over everything. It would catch a `FunctionClauseError`, wrap it in a `try/catch`, show a flash message saying "connection successful," and move on. The QA agent would see the flash message and mark the scenario as passing. Both agents were collaborating to produce a Potemkin village of passing tests over broken functionality.

The Google provider architecture is the clearest example. The factory initially generated a single "Google" provider that handled Ads, Analytics, and Search Console through one OAuth flow. That's architecturally wrong — each Google product has different scopes, different account selection flows, and different API endpoints. Commit `097dbd3` (split Google OAuth into separate providers) is a massive refactor that I had to guide step by step. The factory couldn't have reached this decision on its own because the spec said "connect Google" and it connected Google.

I eventually sat down for a day or two and basically QA'd the integrations with the agent by hand, pointing at each broken flow and walking it through the fix. That was the only way to get working integrations.

## Lessons Learned

**1. QA must be gate-keeping, not advisory.** The factory initially treated QA results as informational — a failed QA run would produce a report, but the pipeline would continue to the next story. I changed the workflow to require all QA scenarios to pass before the agent could move on. But even that wasn't enough, because the agents learned to make QA pass rather than making the code work. The real fix is QA scenarios that test outcomes, not UI elements — "did the OAuth token exchange actually produce a valid access token?" not "does the success flash message appear?"

**2. Multi-tenancy and integrations should be procedurally generated, not LLM-generated.** These are solved problems. Multi-tenancy code is the same everywhere — it's `phx.gen.auth` for accounts. OAuth integrations are the same everywhere — it's request/response/token-storage with provider-specific scopes. I've since written generators for both. An LLM adds variance where you want consistency, and integrations are exactly the domain where variance kills you.

**3. Secrets and external dependencies must be validated before code generation begins.** When the factory generates a QuickBooks integration, it needs to know up front that there's a working sandbox app with valid client ID and secret, and that the redirect URI is configured. When it discovers these problems mid-generation, it papers over them. The human needs to do the OAuth app setup, validate that tokens work, and hand the factory a known-good configuration. Let the machine generate code against a working foundation, not fight two problems at once.

**4. The factory produced a working application — with caveats.** The bulk of MetricFlow works. Authentication, account management, invitations, the dashboard, correlation analysis, data sync scheduling, the QA infrastructure itself — all of that came off the line functional. The factory's output is best understood as a very thorough first draft: architecturally sound, broadly functional, but with integration points and data-dependent features that need human verification. 90% of what it produced worked. The remaining 10% would never have shipped without a human sitting down and clicking through it.
